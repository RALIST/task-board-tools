# TB-107: Show global agent quota in app header

## Goal

Show each registered agent's current global quota usage (from `claude /usage` and `codex /status`) in the GUI app header, so the user can see at a glance how much capacity is left before assigning more tasks to that agent.

## Context

- Scope is **global per agent CLI**, not per task and not per run. The user wants a single header indicator per registered agent (currently `claude` and `codex`) that answers "how many more tasks can I throw at this agent before it hits a quota wall?".
- Data source is the agents' own quota commands:
  - Claude Code reports session/plan usage via the interactive `/usage` slash command (percent of plan consumed, and reset time).
  - Codex CLI reports usage via the interactive `/status` slash command (percent of weekly/5h windows consumed).
- Both commands today are interactive (typed into the agent's TUI). Programmatic access is not obvious — part of this task is to figure out the right invocation path (a non-interactive flag, capturing the same output during a real run, parsing the agent's local config/cache, or a thin wrapper). The investigation outcome must be recorded in the Log.
- Display target is the **app header**, not the TaskDrawer. The header is the right surface because the value is agent-global and changes independently of any individual task.
- Existing per-run token output (Codex's `tokens used` trailer, etc.) is out of scope for this task. If we later want per-run token accounting, it gets its own task.
- Related agent infrastructure: @gui/internal/agent/exec.go, @gui/internal/agent/state.go, @gui/app/agent_run.go, @gui/frontend/src/lib/stores/runs.ts. Header lives in @gui/frontend/src/lib/components/ (top-level layout component, not TaskDrawer).

## Constraints

- Do not burn paid quota on probes. If the only way to get the value is to start a real agent session, cache the latest value seen during a real task run rather than spinning up extra processes.
- Treat each agent's output format as unstable. Per-agent parsers must be fixture-driven, with graceful fallback to "unknown" on missing/malformed/partial output.
- Missing data is normal — agent not installed, not authenticated, plan info unavailable, or never run yet — and must render as "unknown" in the header without errors, broken layout, or fake zeroes.
- One backend-normalized model. The frontend consumes a single `{ agent, percent | null, lastUpdated, source }` shape per agent; no Claude/Codex parsing in Svelte.
- Backward compatibility: existing JSONL run history, TaskDrawer, and past-runs UI stay unchanged. This task adds a separate global usage channel; it does not modify the per-run schema.
- Refresh model: a periodic backend refresh plus a manual refresh affordance in the header. Refresh must not block the UI and must not error-spam if the agent CLI is unreachable.

## Acceptance Criteria

- [ ] The GUI app header shows one compact entry per registered agent (at least `claude` and `codex`) with the latest known percent of quota used and a "last updated" timestamp; entries are visible from every screen.
- [ ] Backend exposes a normalized usage value per agent (percent + lastUpdated + source), populated by parsing the agent CLI's own usage report (`claude /usage`, `codex /status`, or whichever non-interactive path is chosen during investigation).
- [ ] Per-agent parsers have fixture coverage for representative `/usage` and `/status` output, including realistic line shapes captured from the real CLIs.
- [ ] When an agent is missing, unauthenticated, or has never produced usage output, the header entry shows `unknown` (not `0%`) and the rest of the header keeps working.
- [ ] Header refreshes on a sensible interval and via a manual refresh action; refresh failures degrade silently to "unknown" with an unchanged previous value if one exists.
- [ ] Frontend store hydrates from a single backend endpoint / Wails binding and updates live; Svelte components do not parse raw CLI output.
- [ ] Go tests cover: parser fixtures per agent, normalization, fallback when output is absent/malformed, and the backend usage endpoint.
- [ ] Frontend tests cover: header store hydration, live update, known-vs-unknown rendering, and manual refresh.
