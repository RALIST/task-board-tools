# TB-97: CLI: tb create defaults to folder form

**Type:** feature
**Priority:** P0
**Size:** M
**Module:** cli
**Tags:** epic-tb93,cli,folder-tasks,create
**Agent:** codex
**AgentStatus:** success
**Branch:** —
**Parent:** TB-93

## Goal

Make `tb create` write new tasks in folder form (`<status>/<ID>/TASK.md`) by default, with the generated task markdown including an empty `## Attachments` section. Keep legacy file-form creation only if it is intentionally exposed behind an explicit flag.

## Context

- @cli/create.go — `cmdCreate` currently writes `<status>/<ID>.md`; this task changes that write path and the created path printed to stdout.
- @cli/board.go — `allocateID`, `.next-id`, status resolution, and board locking invariants must remain intact.
- @cli/templates.go, @cli/README.md, @board/CONVENTIONS.md — CLI-facing docs/help/templates must describe the new create default and any legacy flag.
- @docs/ARCHITECTURE.md — source of truth for the folder-form contract from TB-94; do not invent a second contract here.
- **TB-93** is the parent epic for folder-backed tasks and attachments.
- **TB-94** should define the exact folder/attachment contract before this task implements it.
- **TB-96** owns read parity for folder-form tasks; sequence after it or coordinate so a freshly created folder task can still be shown, listed, and regenerated.

## Constraints

- CLI-only implementation plus required CLI-facing docs/tests; no GUI attachment workflow in this task.
- Preserve structured mutation invariants: `tb create` holds `.board.lock`, `.next-id` is updated atomically through the existing allocation path, task markdown writes use `writeFileAtomic`, and `BOARD.md` is regenerated through the existing command flow.
- Do not migrate existing file-form tasks, auto-promote legacy tasks, or implement `tb attach`; those belong to TB-99/TB-100.
- The empty `## Attachments` section must be part of newly generated task markdown, but actual attachment discovery/listing semantics stay aligned with TB-94/TB-96.
- If legacy creation remains available, it must be opt-in and obvious (for example `--legacy-file`), documented in help/docs, and covered by tests. Otherwise remove the legacy create path entirely.

## Acceptance Criteria

- [x] Default `tb create "Title"` creates `<status>/<ID>/TASK.md`, does not create a sibling `<status>/<ID>.md`, prints the folder-task path, and still respects `--status`, `--parent`, `--epic`, metadata flags, and description input.
- [x] Generated task markdown includes `## Goal`, `## Acceptance Criteria`, empty `## Attachments`, and `## Log` sections in the documented order.
- [x] ID allocation and mutation safety remain unchanged: `.next-id` is updated atomically while `.board.lock` is held, task markdown writes use `writeFileAtomic`, parent `## Subtasks` updates still work, and `BOARD.md` is regenerated after create.
- [x] If legacy file-form creation is kept, it lives behind an explicit flag such as `--legacy-file`; `tb create --help`, `board/CONVENTIONS.md`, `cli/README.md`, and init templates document both the folder default and the legacy escape hatch.
- [x] `cli/` tests cover the default folder path and, if retained, the legacy flag path; tests assert on-disk layout, generated sections, stdout path, parent-subtask behavior, and no default file-form write.
- [x] `cd cli && go test ./...` passes.
