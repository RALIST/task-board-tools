## Goal

TB-93 is the tracking epic for adding folder-backed tasks and attachments without breaking existing file-backed boards.

Deliver a coexistence model where task identity, status, metadata, board rendering, JSON output, watcher behavior, and agent behavior are the same whether a task is stored as the legacy file form (`<status>/<ID>.md`) or the new folder form (`<status>/<ID>/TASK.md`). Existing file tasks must remain readable and writable in place; they are promoted to folder form only through an explicit operation such as adding an attachment.

Folder tasks own their related files inside the task directory: task markdown, user attachments under a documented path, and per-task agent logs/state under documented task-local paths. Moving, starting, completing, closing, archiving, or restoring a folder task must move the whole task directory under `.board.lock` so artifacts travel with the task and no board-root attachment/artifact directories become the source of truth for folder tasks.
Task itself should contain `Attachments` section with list of task attachments

This epic must be delivered through child tasks rather than as one XL implementation change. It closes only after the CLI read model, CLI write/promotion model, attach/remove commands, move/archive semantics, agent artifact relocation and stale recovery, GUI attachment workflow, watcher behavior, docs, migration notes, and mixed-board smoke test have all shipped and passed their own verification. The GUI must call `tb` for attachment mutations and must not write task files or attachment files directly.

## Acceptance Criteria

- [x] `docs/ARCHITECTURE.md` defines the folder-task contract before implementation: exact folder path, canonical markdown filename (`TASK.md` unless explicitly changed), `attachments/` location, per-task agent log/state paths, file-task legacy paths, lock/atomicity rules, and file-to-folder promotion rules.
- [x] `docs/FEATURES.md` lists TB-93 as an epic/milestone and links the child tasks needed to ship it; each child task has its own goal, acceptance criteria, target surface, and verification command.
- [x] TB-93 is split into child tasks covering at minimum: CLI discovery/read parity for folder tasks; CLI create/write/move/archive semantics; `tb attach` add/remove; file-to-folder promotion; per-task agent logs/state plus stale recovery; `BOARD.md`/JSON/watcher parity; GUI attachment list/open/picker; GUI drag-and-drop; docs/migration notes; final mixed-board smoke.
- [x] CLI read commands (`tb ls`, `tb show`, `tb grep`, `tb triage`, `tb board`, `tb regenerate`, and all relevant `--json` outputs) return the same logical results for all-file, all-folder, and mixed boards, with regression tests in `cli/`.
- [x] `tb create` has a documented storage default. The target default is folder form (`<status>/<ID>/TASK.md`); if legacy file creation remains supported, it is exposed through an explicit documented flag and tested separately.
- [x] `tb attach <ID> <path>...` copies or imports files into the task's documented attachment directory under `.board.lock`; attaching to a legacy file task auto-promotes it to folder form atomically and preserves metadata, log history, and board rendering.
- [x] Attachment removal is available through a documented `tb` command shape such as `tb attach --rm <ID> <attachment>...`; removal runs under `.board.lock`, validates paths to prevent deleting outside the task folder, and is covered by success and failure-path tests.
- [x] `tb mv`, `tb start`, `tb done`, `tb close`, archive, and restore move folder tasks as whole directories, including attachments and colocated agent artifacts; forced-fault tests prove no orphaned task files, attachments, or agent artifacts are left in the source, destination, or board-root agent directories.
- [x] For folder tasks, daemon enqueue, in-flight writes, log retrieval, crash/stale recovery, and any agent runtime cleanup use the task-local agent paths; file tasks keep the existing board-root `.agent-state/` and `.agent-logs/` behavior. Daemon/backend tests cover both forms and mixed boards.
- [x] `BOARD.md` generation is byte-identical for the same logical board regardless of storage form, unless a deliberate path-visible difference is documented in the architecture and snapshot-tested.
- [x] GUI task drawer lists attachments with name, size, and open-in-OS action; adding through file picker and removing through the drawer call `tb` commands rather than writing or deleting files directly.
- [x] GUI drag-and-drop onto a task card and onto the task drawer adds attachments through `tb`, shows success/failure feedback, and refreshes the task from the board after mutation.
- [x] GUI fsnotify/watcher behavior emits one logical refresh for attachment add/remove and folder-task moves; tests or a documented smoke check prove temp files and intermediate renames do not cause visible event storms.
- [x] Final manual smoke covers one mixed board: create folder task, create/read legacy file task, attach via CLI, auto-promote legacy task on attach, attach via GUI picker, attach via GUI drag-and-drop, run agent on a folder task, move across all columns, archive/restore, regenerate `BOARD.md`, and verify no orphaned attachments or agent artifacts remain.
- [x] TB-93 closes only after every linked child task is done and the final mixed-board smoke evidence is recorded on the epic.
