# TB-59: pidAlive(pid, name) probe with command-name cross-check (R10 mitigation)

**Type:** feature

**Module:** gui
**Tags:** milestone-m5,agent,daemon,pid-check
**Branch:** —
**Parent:** TB-5

## Goal

Provide a helper `pidAlive(pid int, expectedAgent string) bool` used by stale-running recovery (TB-60). Returns true only when the PID is still alive *and* the running process's command name (or argv if it's a `node` shebang wrapper) matches what we expect (`claude` or `codex`). Mitigates R10 (PID reuse after crash) so we don't synthesize a `finished{failed}` event against an unrelated process that happens to have the same PID.

## Context

Risk R10 in `docs/IMPLEMENTATION.md`: a process exits, the OS reuses the PID for an unrelated process, and the daemon's recovery check sees "PID alive" and skips recovery — leaving the task stuck in `running` forever. The mitigation: also check the command name.

Two real complications:

1. **`claude` and `codex` are typically npm-installed shebang scripts**, so `ps -o comm=` returns `node` (or `node-22`), not `claude` / `codex`. A naive `comm` basename match would either always fail (recovery never trusts the PID and over-recovers) or always pass (`node` matches nothing distinctive). Use a two-step check:
   - Read `comm` via `ps -o comm= -p <pid>`. If basename equals `expectedAgent` exactly, accept.
   - Otherwise read `args` via `ps -o args= -p <pid>` and accept if the args contain a path whose basename matches `expectedAgent` (e.g. `node /usr/local/bin/claude -p ...`).
2. **`comm` vs `args` semantics differ between macOS and Linux**: macOS `ps -o comm=` returns the full executable path (truncated to 16 chars on some versions); Linux returns the basename. Normalize via `filepath.Base(strings.TrimSpace(out))`.

Liveness check itself: `os.FindProcess(pid).Signal(syscall.Signal(0))` — on POSIX, signal 0 doesn't deliver anything but returns `syscall.ESRCH` if the process doesn't exist and `syscall.EPERM` if it does but is owned by someone else. Treat ESRCH as "dead", EPERM as "alive but not ours" (still alive — caller treats this as "leave alone").

Match semantics: **exact basename match** for `comm` path. `claude` matches `claude` exactly; does not match `claude-bin` or `claude2`. `args`-fallback uses exact basename of any whitespace-separated token. The original "prefix match" wording contradicted the "must not match claude2" carve-out and is dropped.

Edge cases:
- PID is 0 or negative → return false.
- `ps` not on PATH → log WARN and return the liveness-only result (don't return false — that would over-recover).

This is invoked only from recovery; not on the hot path.

## Acceptance Criteria

- [ ] New file `gui/internal/daemon/pid.go` exports `func pidAlive(pid int, expectedAgent string) bool`
- [ ] Returns `false` for `pid <= 0`
- [ ] Liveness via `os.FindProcess(pid).Signal(syscall.Signal(0))`; `errors.Is(err, syscall.ESRCH)` → dead → false; `nil` or `errors.Is(err, syscall.EPERM)` → alive → proceed to name check
- [ ] Name check step 1: `exec.Command("ps", "-o", "comm=", "-p", strconv.Itoa(pid)).Output()`; `filepath.Base(strings.TrimSpace(out))` exact-match against `expectedAgent` → true
- [ ] Name check step 2 (fallback for npm-installed `claude`/`codex` wrappers run via `node`): `exec.Command("ps", "-o", "args=", "-p", strconv.Itoa(pid)).Output()`; split on whitespace; if any token's `filepath.Base` exact-matches `expectedAgent` → true
- [ ] No prefix/substring match: `claude` does **not** match `claude-bin`, `claude2`, or `myclaude`
- [ ] If `ps` is unavailable (LookPath fails), log WARN and return the liveness result (alive→true, dead→false) — don't over-recover
- [ ] Unit test (with real spawned child processes): live PID + matching `comm` → true; live PID + `comm`=`node` and `args` contains a path basenamed `claude` → true; live PID + unrelated name → false; dead PID → false; negative PID → false; `claude2` does not match `claude` → false
- [ ] Cross-platform note: macOS + Linux only (Windows out of MVP scope per `docs/ARCHITECTURE.md` "OS support")
