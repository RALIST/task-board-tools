## Goal

Define the folder-task on-disk contract before any implementation work begins so all child tasks share one source of truth.

## Context

- @docs/ARCHITECTURE.md — primary edit target. Existing sections to extend or cross-link: "On-disk layout", "Task file format", "Locking and atomic writes", "Agent state".
- @docs/FEATURES.md — must gain a cross-reference once the new section lands (TB-95 builds on this).
- @CLAUDE.md — "Architecture invariants" bullet list must point to the new section by name.
- @board/CONVENTIONS.md — current task file-format reference (file-form `<status>/<ID>.md`); the new spec describes its folder-form counterpart.
- @cli/atomicfs.go — sanctioned write path (`writeFileAtomic`); the spec must keep this invariant for files inside a task folder.
- Parent epic body (`tb show TB-93`) — describes the coexistence model and names `TASK.md` / `attachments/` as canonical paths.

## Constraints

- Documentation-only change: no edits to code, tests, configuration, generated files, or assets.
- Preserve existing invariants: markdown is source of truth, `.board.lock` serializes every structured mutation, every task-file write goes through `writeFileAtomic`.
- Canonical folder filename is `TASK.md`; canonical attachments path is `<status>/<ID>/attachments/`. Per-task agent log/state paths must be task-local (inside the folder) for folder tasks, while file tasks keep the board-root `.agent-state/` / `.agent-logs/` behavior.
- Describe file-form (`<status>/<ID>.md`) and folder-form (`<status>/<ID>/TASK.md`) side by side so a reader can compare layouts without inferring.
- The file → folder promotion procedure must be explicit, atomic, and lock-held; spell out the rename/copy ordering so partial promotions are impossible.
- Any deliberate path-visible difference between forms (e.g. `BOARD.md` location) must be called out explicitly with a one-line rationale.
- Reference style: `docs/FEATURES.md` and `CLAUDE.md` should anchor to the new section by name so future readers find it from either entry point.

## Acceptance Criteria

- [ ] `docs/ARCHITECTURE.md` gains a "Folder-form tasks" section defining: exact directory layout, canonical markdown filename (`TASK.md`), `attachments/` location, per-task agent log/state paths, and file-task legacy paths.
- [ ] Section spells out lock semantics (`.board.lock` scope), atomic-write / temp-rename rules for files inside a task folder, and the file → folder promotion procedure.
- [ ] Section identifies any deliberate path-visible difference between file and folder forms (e.g. for `BOARD.md`).
- [ ] Section is referenced from `docs/FEATURES.md` and from `CLAUDE.md` → "Architecture invariants" so it cannot be missed when implementing children.

## Done when

- All four acceptance criteria above are checked.
- `docs/ARCHITECTURE.md`, `docs/FEATURES.md`, and `CLAUDE.md` have been edited in the same change so the cross-references resolve.
- No code, test, or configuration file is modified.
