# TB-62: Graceful shutdown: ctx cancel + 5s grace + JSONL flush

**Type:** feature
**Priority:** P1
**Size:** M
**Module:** gui
**Tags:** milestone-m5,agent,daemon,shutdown
**Branch:** —
**Parent:** TB-5

## Goal

When the GUI is closed, `Daemon.Close()` cancels the daemon's root context, waits up to 5 seconds for in-flight runners to flush their JSONL `finished` events, then returns. If the grace expires, return anyway — the next-start recovery (TB-60) will reconcile whatever state was left behind.

## Context

F5.4: "App close cancels context, waits 5s for runners to flush, then exits." Acceptance: "close GUI during a run; `.agent-state` file has a coherent `finished` event (status either success/failed/cancelled, not orphaned)."

**Critical prerequisite from TB-54**: today `gui/app/agent_run.go:148` creates the runner ctx as `context.WithCancel(context.Background())`, *not* as a child of any caller-supplied ctx. As-is, `Daemon.Close()` cancelling its root ctx will not reach any in-flight runner — the 5s grace would be a no-op. TB-54 adds an internal executor (`RunQueuedAgentSync(ctx, id)`) that uses the **caller-supplied ctx** as the runner-ctx parent. This task depends on that plumbing and asserts it works.

When the daemon cancels its ctx, the child process receives whatever `runGoroutine` translates `ctx.Done` into — currently SIGKILL via `exec.CommandContext`'s default. The M4 cancel path uses a pgid-aware SIGTERM-then-SIGKILL escalation (`agent_cancel.go`) which is what we want here, not the default kill. Factor the cancel-finish work — JSONL `finished{cancelled}` write, Wails emit, `tb edit --agent-status cancelled` — into a private helper `(s *AgentService).finishCancelled(ar, reason string)` that both `CancelRun` (`reason: "user"`) and the daemon-shutdown path (`reason: "shutdown"`) call. The escalation policy itself stays in `CancelRun`.

Sequence on shutdown:
1. `cancel()` — propagates to all workers' ctxs.
2. Workers' active runs hit ctx-cancelled inside their copy loop; the existing M4 post-run handler writes `finished{status: cancelled, reason: "shutdown"}` and `tb edit --agent-status cancelled`.
3. Daemon waits on a `sync.WaitGroup` for all workers to return, with a 5s timeout.
4. On timeout: log WARN with count of still-running workers and return — leftover state will be recovered by TB-60 on next start.

Caveat: the post-run handler does `tb edit` via shell-out, which can take 50-200ms. With many concurrent runs the 5s budget may be tight. The 5s is a documented soft cap; if it proves too short during testing, bump to 10s and update the docs.

`gui/main.go` wires `daemon.Close()` after `app.Run()` returns. Wails3 alpha may have its own `OnShutdown` hook — prefer that if it fires before the asset server is torn down.

## Acceptance Criteria

- [ ] **TB-54 prerequisite verified**: `AgentService.RunQueuedAgentSync(ctx, id)` uses the caller-supplied ctx as the runner ctx parent (replacing today's `context.Background()` at `agent_run.go:148`). This task fails to ship without that plumbing.
- [ ] `AgentService` exposes a private cancel-finish helper that takes a `reason` parameter (e.g. `"user"`, `"shutdown"`); both `CancelRun` and the daemon-shutdown path call it. `CancelRun` keeps its 5-step escalation; the helper just owns the post-kill JSONL+emit+`tb edit`.
- [ ] `Daemon.Close() error` cancels the root context, then waits on a `sync.WaitGroup` with a 5s timeout
- [ ] Workers, on ctx-cancelled, run the cancel-finish helper with `reason: "shutdown"` so the JSONL line is distinguishable from a user cancel
- [ ] `gui/main.go` calls `daemon.Close()` after `app.Run()` returns (or in Wails3 `OnShutdown` if available); ignores returned error but logs it
- [ ] **F5.4 acceptance**: close the GUI window during an active run; `.agent-state/<ID>.jsonl` ends with `finished{status: cancelled, reason: "shutdown"}`; `tb show X` reports `AgentStatus=cancelled`; restart GUI → recovery (TB-61) leaves the task cancelled
- [ ] If grace expires with workers still running: log WARN `shutdown grace expired, N workers still active` and return; next-start recovery (TB-60) handles the leftover state; assert no panic / no goroutine leak in tests via `goleak.VerifyNone(t)`
- [ ] Integration test: start daemon with a fake runner that sleeps 10s; call Close; assert it returns within ~5.5s and the JSONL has a `finished{status: cancelled, reason: "shutdown"}` line
