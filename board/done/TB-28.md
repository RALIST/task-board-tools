# TB-28: collectAllTasks / findChildren: archive inclusion semantics

**Type:** improvement
**Priority:** P3
**Size:** M
**Module:** cli
**Tags:** archive,filters
**Branch:** main

## Goal

Make archive visibility intentional across the derived board views instead of an accident of `statusDirs`.

## Context

M1 made `archive` a first-class status for explicit filters: `tb ls --status archive` reads only archived tasks and `tb ls --status all` reads active plus archived tasks. The remaining derived views still use helpers whose names imply "all" while they actually scan only `backlog`, `in-progress`, and `done`:

- `buildBoardContent` / `tb board` use `collectAllTasks` for epic lists and progress, then `collectTasks("done")` for Recently Done.
- `tb board --json` mirrors that same active-only shape through `buildBoardSnapshot`.
- `tb epic` finds the epic itself in active or archive via `findTask`, but `findChildren` scans only active status directories.

That behavior may be correct, but it is currently implicit and easy to break while extending archive support.

## Scope Decision

Default board surfaces stay active-only:

- `BOARD.md`, text `tb board`, and `tb board --json` do not list archived epics or archived children.
- Recently Done remains the current `done/` directory only; archived closed tasks do not reappear there.
- Epic progress counts only active-scope children and treats only current `done` children as complete.

Explicit archive/all inspection belongs behind explicit status filtering. If `tb epic` gains archive visibility in this task, it must be opt-in with a `--status active|archive|all` flag using the same resolver as `tb ls`; the default remains active.

## Acceptance Criteria

- [x] `collectAllTasks` is either renamed/split or documented so callers cannot confuse active-board scope with `--status all` scope.
- [x] `buildBoardContent` and `buildBoardSnapshot` have tests proving archived epics and archived child tasks do not appear in default board output or board JSON.
- [x] Recently Done has a regression test proving it includes `done/` tasks only and excludes `archive/` tasks.
- [x] `findChildren` is aligned with the same status-scope decision: default `tb epic <ID>` lists/counts active children only, or an explicit `tb epic --status ...` path is added and tested.
- [x] If `tb epic --status archive|all` is added, help text and sorting handle `[archive]` children explicitly; progress still counts only current `done` children as done.
- [x] Docs or CLI comments explain that `archive` is a closed/hidden status, not a synonym for `done`.
- [x] `cd cli && go test ./...` passes.

## Non-Goals

- Do not add an Archive section to `BOARD.md`.
- Do not change `tb ls` / `tb grep` status filter semantics from TB-14.
- Do not infer an archived task's previous status from log text.
