# TB-90: Board switching is not working

**Type:** bug
**Priority:** P0
**Size:** S
**Agent:** codex
**AgentStatus:** success
**Branch:** —

## Goal

Fix the remaining GUI board-switching bug in the header `Open board...` flow: selecting a board that was already opened earlier in the same app session must immediately refresh the running UI, not only persist the selection for the next app reload.

Repro:
- Start the GUI with `wails3 dev` from `/Users/ralist/projects/task-board-tools`.
- Confirm the app initially displays the `task-board-tools` board.
- Use header `Open board...` and choose `/Users/ralist/projects/books/writer-studio`.
- Confirm the visible board changes to Writer Studio.
- Use header `Open board...` again and choose `/Users/ralist/projects/task-board-tools`.

Current behavior: the selected path is apparently saved because reloading the app shows the correct `task-board-tools` board, but the running UI still shows the previously active Writer Studio board until reload.

Expected behavior: every valid board selection becomes the active board immediately in the current app session, including switching back to a previously opened board path. Header/project identity, columns, and task list must all reflect the newly selected board without restarting or reloading the app.

Scope: only the header board picker and board-switch/reload state path. Do not change the markdown board format, task parsing, task rendering semantics, or unrelated navigation behavior.

## Acceptance Criteria

- [x] From a fresh `wails3 dev` session in `/Users/ralist/projects/task-board-tools`, the GUI opens with the `task-board-tools` board visible in the header/project identity, columns, and task list.
- [x] Selecting `/Users/ralist/projects/books/writer-studio` through the header `Open board...` action changes the active board and all visible board data to Writer Studio without app reload.
- [x] Immediately selecting `/Users/ralist/projects/task-board-tools` through the same `Open board...` action changes the active board and all visible board data back to `task-board-tools` without app reload; no Writer Studio header, column, or task data remains visible.
- [x] Repeating the two-path switch sequence at least twice in one app session works every time, including paths that were already opened earlier in that session.
- [x] Verification covers the live switch-back path specifically: added a focused frontend store test that simulates the task-board-tools -> Writer Studio -> task-board-tools switch-back with the older Writer Studio refresh completing late.
