# TB-90: Board switching is not working

**Type:** bug
**Priority:** P0
**Size:** S
**Agent:** codex
**AgentStatus:** success
**Branch:** —

## Goal

Fix the GUI board-switching regression in the header "Open board..." flow.

Repro:
- Start the GUI with `wails3 dev` from `/Users/ralist/projects/task-board-tools`.
- The app initially opens the `task-board-tools` board.
- Click `Open board...` in the header and choose `/Users/ralist/projects/books/writer-studio`.
- The app switches to and displays the Writer Studio board.
- Click `Open board...` again and choose `/Users/ralist/projects/task-board-tools`.

Current behavior: the second switch is ignored and the UI continues to show the Writer Studio board.

Expected behavior: each selected board path becomes the active board immediately, including switching back to a board that was open earlier in the same app session.

Scope: board selection/switching behavior only; do not change the markdown board format or unrelated task rendering behavior.

## Acceptance Criteria

- [x] From a fresh `wails3 dev` session in `/Users/ralist/projects/task-board-tools`, the GUI opens and displays the `task-board-tools` board.
- [x] Using the header `Open board...` action to select `/Users/ralist/projects/books/writer-studio` changes the active board and visible board content to Writer Studio without restarting the app.
- [x] Using the same `Open board...` action to select `/Users/ralist/projects/task-board-tools` immediately afterward changes the active board and visible board content back to `task-board-tools`; Writer Studio content is no longer shown.
- [x] Alternating between the two board paths at least twice in one app session works every time, with no stale header, task list, or column data from the previous board.
- [x] The fix is covered by the narrowest practical automated test for the board-switching state/reload path, or by a documented GUI smoke check if the behavior cannot be exercised below the Wails UI layer.
