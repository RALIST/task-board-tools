## Goal

Make the GUI agent subsystem resolve agent state and log artifacts by task storage form: folder tasks use the task-local paths defined by the folder-task contract, while legacy file tasks keep the board-root `.agent-state/` and `.agent-logs/` layout.

## Context

- @board/backlog/TB-93.md ‚Äî parent epic; requires folder tasks to own task-local agent logs/state and requires file tasks to remain backward compatible.
- @board/backlog/TB-94.md / @docs/ARCHITECTURE.md ‚Äî prerequisite contract for exact folder-task paths. Do not invent a second path layout here.
- @gui/internal/agent/state.go ‚Äî current source of `AppendEvent`, `NewLogWriter`, `StatePath`, and `LogPath`; today these always target board-root `.agent-state/` and `.agent-logs/`.
- @gui/app/agent_run.go and @gui/app/agent_finish.go ‚Äî daemon enqueue, queued-run lookup, in-flight writes, terminal writes, and log writer setup.
- @gui/app/agent_runs.go ‚Äî `ListRuns` and `GetRunLog` must read from the same resolved artifact location used by writers.
- @gui/app/agent_recovery.go and @gui/app/daemon_integration_test.go ‚Äî stale-running recovery and daemon pickup tests, including PID-liveness behavior.

## Constraints

- Preserve file-form task compatibility exactly: `<board>/.agent-state/<ID>.jsonl` and `<board>/.agent-logs/<ID>/<runID>.log` remain valid for legacy file tasks.
- Folder-form tasks must not write new agent state or logs into board-root agent directories; artifacts live inside `<status>/<ID>/` according to TB-94.
- Keep JSONL event schema, Wails event names, run ordering, and `AgentStatus` lifecycle semantics unchanged.
- Recovery must preserve the existing cancelled carve-out: never turn durable `cancelled` state into `failed`.
- Do not implement attachment commands, task promotion, folder moves/archive, or watcher coalescing here; those belong to TB-98, TB-99, TB-103, TB-104, and TB-105.

## Acceptance Criteria

- [x] Agent artifact path resolution is folder-aware and shared by all writers/readers instead of duplicating path decisions at call sites.
- [x] For folder-form tasks (`<status>/<ID>/TASK.md`), daemon enqueue, queued-run lookup, `AppendEvent`, `NewLogWriter`, `ListRuns`, `GetRunLog`, terminal writes, and stale recovery use the task-local state/log paths defined by TB-94.
- [x] For file-form tasks (`<status>/<ID>.md`), all existing run history and log behavior continues to use board-root `.agent-state/<ID>.jsonl` and `.agent-logs/<ID>/<runID>.log`.
- [x] Mixed boards can run, list, read logs, and recover stale runs for one folder task and one file task without cross-writing artifacts into the wrong layout.
- [x] Stale-running recovery walks both layouts, still uses the PID-liveness probe, synthesizes `failed` only for dead unfinished runs, and never overwrites a durable `AgentStatus=cancelled` / `finished{cancelled}` outcome.
- [x] Backend tests cover folder-form, file-form, and mixed-board cases for run writes, log retrieval, daemon queued pickup, stale recovery, live-PID skip, and cancelled carve-out behavior.

## Manual test (UI)

Open the GUI on a board containing one folder task and one file task. Run an agent on the folder task, open the past-runs/log view, and confirm state/log artifacts appear only inside that task folder. Start another folder-task run, stop the daemon mid-run, restart the GUI, and confirm recovery reconciles the running task without overwriting a cancelled outcome. Repeat a normal run and log read on the file task and confirm legacy board-root `.agent-state/` and `.agent-logs/` paths still work.
