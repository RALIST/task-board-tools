# TB-71: Frontend api.ts + triageStore.ts: groomTask, getTriage, watcher-driven refresh

## Goal

Add typed frontend wrappers for `AgentService.GroomTask` and `BoardService.Triage`, plus a `triageStore.ts` Svelte 5 store keyed by task ID. The store auto-refreshes on the existing `board:reloaded` Wails event (the same event the kanban already listens to) so cards see the latest "needs grooming" set without per-card requests.

## Context

`api.ts` already shells the M4 / M5 service surfaces (`AssignAgent`, `RunAgent`, `CancelRun`, `ListRuns`); this just adds two more methods that mirror that shape. The store pattern follows `runsStore.ts` (TB-52) — a Svelte 5 `$state` rune holding a `Map<taskID, string[]>` plus subscribers. The agent-side event types also need to grow: `agent:run-queued`/`-started`/`-finished` payloads gain `mode: 'implement' | 'groom'` (TB-67 emits it; the frontend just types it).

The store must mirror the backend cache invalidation (TB-70): `board:reloaded` clears + reloads the full map; `task:updated:<id>` refreshes that single ID. Without the per-ID refresh, a card whose task was edited via inline metadata changes would keep a stale "needs grooming" badge until a full board reload. The watcher already emits `task:updated:<id>` for in-place metadata writes (priority/module/agent/agent-status edits) — see `gui/internal/watcher/watcher.go` for the event shape.

## Acceptance Criteria

- [ ] `gui/frontend/src/lib/api.ts` exports `groomTask(id: string): Promise<string>` (returns run_id, same shape as `runAgent`)
- [ ] `gui/frontend/src/lib/api.ts` exports `getTriage(): Promise<Record<string, string[]>>`
- [ ] `AgentEvent` / `AgentRun` types gain a `mode: 'implement' | 'groom'` field; existing call sites default to `'implement'` for backwards-compat
- [ ] New `gui/frontend/src/lib/stores/triage.ts` exports a `triageStore` (Svelte 5 rune-based) with `subscribe`, `needsGrooming(id)`, and `reasonsFor(id)` accessors
- [ ] Store seeds itself on first subscribe (single `getTriage()` call) and refreshes the full map on `board:reloaded`
- [ ] Store also subscribes to `task:updated:<id>` and refreshes the per-ID entry (calls `getTriage()` and replaces just that key — relies on TB-70's per-ID cache update) so cards reflect inline edits without a full board reload
- [ ] When `task:updated:<id>` fires for a task that is no longer flagged, the store removes the entry so the indicator disappears immediately
- [ ] Vitest covers: store starts empty, seeds on subscribe, replaces map on `board:reloaded`, per-ID refresh on `task:updated:<id>` (add / replace / remove cases), exposes `needsGrooming` correctly
- [ ] No backend change in this task — only types + frontend wiring
