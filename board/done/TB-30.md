# TB-30: tb assign sugar for paired Agent + AgentStatus writes

**Type:** feature
**Priority:** P3
**Size:** S
**Module:** cli
**Tags:** agent,ergonomics
**Branch:** тАФ

## Goal

Add `tb assign <ID> <agent>` as a CLI convenience for delegating a task to the daemon: one locked mutation sets `Agent=<agent>` and `AgentStatus=queued`, regenerates `BOARD.md`, and avoids the current split-field foot-gun where `tb edit -a claude` does not queue anything and `tb edit --agent-status queued` can leave an un-runnable task with no agent.

## Context

M1 shipped `Agent` / `AgentStatus` metadata and the low-level `tb edit -a <agent> --agent-status <status>` path (TB-11). M4 uses `tb edit -a` for GUI assignment without queueing (TB-46), and M5 only auto-picks up tasks when both fields are present: `AgentStatus=queued` and `Agent != ""` (TB-57/TB-58). Current `tb edit` behavior is intentionally flexible for lifecycle transitions, but it is awkward for the common terminal action "give this task to claude/codex now".

The CLI must keep the existing hybrid ownership boundary from `docs/ARCHITECTURE.md`: task metadata lives in the `.md` file and is safe for CLI mutation; `.agent-state/*.jsonl` and `.agent-logs/*` are GUI/daemon run-history artifacts. M5 already handles CLI-originated queued tasks with no JSONL queued event by synthesizing that event inside `RunQueuedAgentSync`.

## Scope Decision

`tb assign` means "assign and queue for daemon pickup", not "just set the Agent dropdown value". The existing `tb edit -a <agent>` / `tb edit -a none` paths remain the explicit way to set or clear assignment without queueing. `tb assign` accepts only runnable agents (`claude`, `codex`) and always sets `AgentStatus=queued`.

## Acceptance Criteria

- [x] Add a new CLI command dispatch path: `tb assign <ID> <agent>` in `cli/main.go`, with top-level usage/help text that describes it as assigning an agent and queueing the task for daemon pickup.
- [x] Implement the command in the CLI package using the same task mutation invariants as `tb edit`: normalize task IDs, acquire `.board.lock`, use `findTask` across active + archive statuses, write via `writeFileAtomic`, append exactly one log entry, and call `regenerateBoard` after the metadata write.
- [x] `tb assign 30 claude` and `tb assign TB-30 codex` both write exactly these metadata values on the target task: `**Agent:** claude|codex` and `**AgentStatus:** queued`.
- [x] The mutation is atomic from the task-card perspective: both fields are written in the same locked read/modify/write operation, not as two independent CLI edits. The log entry names both applied values in one entry.
- [x] Existing `tb edit` semantics are unchanged: `tb edit <ID> -a claude` still sets only `Agent`; `tb edit <ID> --agent-status queued` still remains available for lifecycle plumbing and validation tests.
- [x] Invalid or missing arguments fail with clear validation errors and non-zero exit: missing ID, missing agent, too many positional args, or an unsupported agent. `none` is rejected for `tb assign`; clearing assignment remains `tb edit -a none [--agent-status none]`.
- [x] `tb assign` does not create or modify `.agent-state/` or `.agent-logs/`; the daemon remains responsible for JSONL queued-event synthesis for CLI-originated queues.
- [x] Add focused CLI tests covering happy path, ID normalization, invalid agent / `none`, no accidental JSONL/log writes, one log entry, and unchanged `tb edit -a` behavior.
- [x] Verification: `cd cli && go test ./...`.

## Non-Goals

- Do not change daemon pickup semantics, worker behavior, stale recovery, or watcher routing.
- Do not change GUI `AgentService.AssignAgent`; the drawer assignment flow should still set only `Agent` until the user clicks Run.
- Do not make the CLI write run-history JSONL or log files.
- Do not add multi-agent assignment, scheduling, priorities, or a new `AgentStatus` value.
