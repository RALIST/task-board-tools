# TB-47: AgentService.RunAgent: enqueue, spawn runner, bridge Wails events

**Type:** feature
**Priority:** P1
**Size:** M
**Module:** gui
**Tags:** milestone-m4,agent,wails-service,wails-events
**Branch:** —
**Parent:** TB-4

## Goal

The hot path. `RunAgent(id)` packages the prompt, picks the right `Runner`, owns the `AgentStatus` lifecycle, streams events to disk and to the frontend, and returns a `run_id` the caller can track. M4 spawns synchronously inside this method; M5 (TB-5) replaces the synchronous spawn with daemon-queue enqueue while keeping every other contract (JSONL events, Wails events, AgentStatus writes) identical.

## Acceptance Criteria

- [ ] `AgentService.RunAgent(ctx, id) (runID string, err error)` — rejects with `ErrNoAgent` if no agent assigned and `ErrAlreadyRunning` if task already has `AgentStatus ∈ {queued, running}`
- [ ] Run ID generated via TB-45 helper (`r_<8 hex>`)
- [ ] **Async return semantics**: `RunAgent` does steps 1–4 below synchronously, then returns the `run_id` and `nil` error. The runner spawn and post-run lifecycle (steps 5–7) run in a background goroutine. The frontend learns about completion through Wails `agent:run-finished` — not through this return
- [ ] **`activeRun` struct** (defined in this task, used by TB-48):
  ```go
  type activeRun struct {
      RunID, TaskID, Agent string
      Mode                 agent.Mode
      Cmd                  *exec.Cmd // populated by OnStarted callback
      Pid, Pgid            int       // populated by OnStarted callback
      Cancel               context.CancelFunc
      Cancelled            bool      // set by TB-48 before kill; checked by post-run handler
      Done                 chan struct{} // closed when post-run handler finishes
      mu                   sync.Mutex
  }
  ```
  Stored in `AgentService.active map[taskID]*activeRun`, guarded by `AgentService.mu`. TB-48 reaches in under `mu`, flips `Cancelled`, then unlocks before killing
- [ ] Sequence on a successful run:
  1. Append JSONL `queued{ts, run_id, task_id, agent, mode: implement}` (TB-45 writer)
  2. Emit Wails `agent:run-queued{run_id, task_id, agent, mode}` so TB-52 can insert a queued row
  3. Set `AgentStatus: queued` via `tb edit --agent-status queued`, then immediately `AgentStatus: running` once the goroutine starts (two writes; the first matters in M5 when there's a real wait between them, in M4 they're ~10ms apart)
  4. Register `activeRun` under `AgentService.mu`, return `runID, nil`
  5. *(goroutine)* Spawn the TB-44 Runner with `OnStarted` callback that populates `Cmd/Pid/Pgid` and `AgentStatus: running`, appends JSONL `started{ts, run_id, task_id, pid}`, emits Wails `agent:run-started{run_id, task_id, agent, mode, pid}`
  6. *(goroutine)* For every line on stdout/stderr: append JSONL `stdout`/`stderr` event AND write to the run log file (TB-45 `NewLogWriter`) AND emit Wails `agent:run-log{run_id, task_id, stream, line}`
  7. *(goroutine, post-run handler)*: under `activeRun.mu`, check `Cancelled`. If true, close `Done` and return — TB-48 owns the cancel-path writes. If false, append JSONL `finished{ts, run_id, task_id, status, exit_code}` (status is `success` for exit 0, `failed` otherwise), emit Wails `agent:run-finished{run_id, task_id, status, exit_code}`, write `AgentStatus: success|failed` via `tb edit`, remove the entry from `AgentService.active`, close `Done`
- [ ] **JSONL ↔ Wails event mapping** (locked here, referenced by TB-45/TB-51/TB-52):
  | JSONL event | Wails event | Payload |
  |-------------|-------------|---------|
  | `queued`    | `agent:run-queued`    | `{run_id, task_id, agent, mode}` |
  | `started`   | `agent:run-started`   | `{run_id, task_id, agent, mode, pid}` |
  | `stdout`/`stderr` | `agent:run-log` | `{run_id, task_id, stream, line}` (`stream ∈ {"stdout","stderr"}`) |
  | `finished`  | `agent:run-finished`  | `{run_id, task_id, status, exit_code}` |
  Every Wails event payload carries `task_id` so frontend consumers (TB-51, TB-52) don't need a runID→taskID index
- [ ] **Error → status mapping** for the post-run handler:
  | Runner outcome | JSONL `finished.status` | `finished.reason` | `AgentStatus` | `exit_code` |
  |----------------|------------------------|-------------------|---------------|-------------|
  | `ExitCode == 0`, `Err == nil` | `success` | — | `success` | `0` |
  | `ExitCode != 0`, `Err == nil` | `failed` | `"non-zero exit"` | `failed` | exit code |
  | `Err == ErrBinaryNotFound` | `failed` | `"binary not found"` | `failed` | `-1` |
  | `Err == ErrTimeout` | `failed` | `"timeout"` | `failed` | `-1` |
  | `Err == context.Canceled` | (cancel path; TB-48 writes `cancelled`) | — | — | — |
  | Any other `Err` | `failed` | `err.Error()` | `failed` | `-1` |
- [ ] Tests with a stub Runner:
  - Emit 10 stdout + 2 stderr lines, exit 0 → JSONL has `queued`+`started`+10×stdout+2×stderr+`finished{success}`; log file has all 12 lines verbatim; Wails sink saw `run-queued`+`run-started`+12×`run-log`+`run-finished`; final `AgentStatus: success`
  - Repeat with exit 1 → `AgentStatus: failed`, `finished.reason: "non-zero exit"`
  - Repeat with `OnStarted` returning `ErrBinaryNotFound` → `AgentStatus: failed`, `finished.reason: "binary not found"`, no `started` JSONL event emitted
  - Concurrent `RunAgent` for the same task ID → second call returns `ErrAlreadyRunning` immediately
