## Goal

fsnotify/watcher must emit a single logical refresh event for attachment add/remove and for folder-task moves; temp files and intermediate renames must not cause visible event storms. Prove via tests or a documented smoke check.

## Context

This is the GUI watcher/coalescing slice of parent epic **TB-93**. It verifies that the folder-task storage work remains calm from the desktop app's point of view after the CLI and GUI attachment tasks land.

Relevant surfaces are @gui/internal/watcher/watcher.go, @gui/internal/watcher/watcher_test.go, @gui/internal/watcher/integration_test.go, @gui/frontend/src/routes/+page.svelte, and the folder-task/attachment commands from **TB-98**, **TB-99**, and **TB-100**. The current watcher watches status directories, emits `board:reloaded` for create/remove/rename bursts, emits `task:updated:<id>` for task writes, and debounces board reloads.

## Constraints

- Preserve the existing logical event contract: folder moves should produce one board-level refresh, while attachment add/remove may produce one task-level or board-level refresh as long as the visible board/drawer refreshes once.
- Keep temp files, `.tmp` atomic-write artifacts, regenerated `BOARD.md`, `.board.lock`, and intermediate rename noise out of the Wails-facing event stream.
- Do not make the GUI write attachment files directly; picker/remove/drop flows remain owned by **TB-103** and **TB-104** and must call `tb`.
- If folder-form tasks require watching nested task directories or `attachments/`, add that coverage without regressing legacy file-form task watcher behavior.

## Acceptance Criteria

- [x] The fsnotify watcher emits exactly one logical refresh event for: attachment add, attachment remove, and a folder-task move across statuses.
- [x] Temp files (e.g. atomic-rename `.tmp` suffixes) and intermediate renames inside a task folder do not bubble visible event storms to the UI.
- [x] Debounce / coalesce logic is covered by watcher tests or an integration test using real `tb` mutations; any manual smoke check records observed event counts.
