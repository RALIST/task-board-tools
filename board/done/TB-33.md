## Goal

Implement direct-write body editor per ARCHITECTURE.md — acquire .board.lock, reject header/metadata mutation, append log entry, writeFileAtomic, exec tb regenerate. The one and only direct-write path in the GUI.

## Acceptance Criteria

- [ ] `BoardService.EditTaskBody(id, newBody)` acquires `.board.lock` via `syscall.Flock(LOCK_EX)` and releases on return
- [ ] Rejects writes that modify the title line (`# PREFIX-NNN: …`) or any of the first 15 metadata lines (returns typed `ErrHeaderMutation`)
- [ ] Appends `- YYYY-MM-DD: Edited body via GUI` to the `## Log` section
- [ ] Writes the new file via the same atomic temp+fsync+rename pattern used by the CLI (no `os.WriteFile` on `.md`)
- [ ] After the rename, execs `tb regenerate` so `BOARD.md` reflects the change
- [ ] Unit tests cover: clean body edit, header mutation rejection, metadata mutation rejection, log entry append, atomic rename verification
- [ ] Real-process POSIX integration test (not mocked): while `EditTaskBody` holds `.board.lock`, spawn `tb edit` as a child process and assert it blocks until the lock is released, then succeeds. Uses the same real-`tb`-binary harness as `gui/internal/watcher/integration_test.go`. This proves we satisfy the flock invariants in `docs/ARCHITECTURE.md` → "Locking and atomic writes"
