# TB-96: CLI: read folder-form tasks identically to file-form

## Goal

Teach CLI read/discovery paths to treat folder-form tasks (`<status>/<ID>/TASK.md`) as the same logical tasks as legacy file-form tasks (`<status>/<ID>.md`) across all read commands and JSON outputs.

## Context

- @cli/board.go — owns status directories, `isTaskFile`, `findTask`, `findTaskInStatuses`, and shared status filter helpers that currently assume `<ID>.md`.
- @cli/task.go — `parseTaskFile` parses metadata and must work when the canonical markdown path is inside a task directory.
- @cli/list.go, @cli/show.go, @cli/grep.go, @cli/triage.go, @cli/regenerate.go, @cli/json_output.go — read surfaces that must use the same folder-aware discovery behavior instead of each hand-scanning files differently.
- @board/backlog/TB-93.md — parent epic defining coexistence between file-form and folder-form tasks.
- @board/backlog/TB-94.md — contract gate for canonical paths (`TASK.md`, `attachments/`) and path-visible differences.

## Constraints

- Read-only scope: do not implement task creation defaults, file-to-folder promotion, attach/remove mutations, folder moves, archive/restore behavior, watcher changes, or GUI changes here. Those belong to TB-97, TB-98, TB-99, TB-100, TB-103, TB-104, and TB-105.
- Introduce or reuse one shared discovery abstraction so `ls`, `show`, `grep`, `triage`, `board`, `regenerate`, epic progress, and JSON snapshots cannot drift by scanning different path shapes.
- A task ID must resolve to exactly one canonical markdown path in a given status scope. If both `<status>/<ID>.md` and `<status>/<ID>/TASK.md` exist, fail loudly instead of picking one.
- Logical parity means ID, title, metadata, status, filtering, sorting, board rows, grep matches, triage reasons, and JSON field semantics match across storage forms. Storage-path fields such as `filePath` may show the actual canonical markdown path and should be documented/tested as path-visible differences.
- Attachment handling in this slice is read/parity only: fixtures may include `<status>/<ID>/attachments/` and an `## Attachments` section listing full attachment paths, but creating/updating that section belongs to create/attach tasks.

## Acceptance Criteria

- [x] Shared task discovery walks both `<status>/<ID>.md` and `<status>/<ID>/TASK.md` in backlog, in-progress, done, and archive scopes; duplicate file/folder forms for the same task ID return a clear non-zero error.
- [x] `findTask`, `findTaskInStatuses`, `collectTasks`, `collectActiveTasks`, and any command-specific scans use the shared discovery path so `tb ls`, `tb show`, `tb grep`, `tb triage`, `tb board`, `tb regenerate`, and `tb epic` all see the same task set.
- [x] `tb ls --json`, `tb show --json`, `tb board --json`, and `tb triage --json` keep the existing camelCase JSON contract while reporting folder-form tasks with the correct status and canonical `filePath`.
- [x] All-file, all-folder, and mixed-board fixtures produce the same logical read results for metadata, sorting, filters, board rows, grep matches, triage reasons, and epic progress; any intentional `filePath` difference is asserted explicitly.
- [x] A folder-form fixture with files under `attachments/` and an `## Attachments` section listing full attachment paths remains readable through `tb show`, `tb show --json`, `tb grep`, and board/list commands without metadata parse drift.
- [x] `parseTaskFile` works from either canonical markdown path and does not infer status incorrectly from the extra folder level.
- [x] `cli/` tests cover all-file, all-folder, mixed, archive-inclusive, duplicate-collision, malformed-folder-task, and attachment-section fixtures.
