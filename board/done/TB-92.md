# TB-92: Limit showed tags in header to 10

**Type:** bug
**Priority:** P1
**Size:** M
**Agent:** claude
**AgentStatus:** running
**Branch:** —

## Goal

Regression: the GUI header tag area still renders more than 10 tag chips, even though a previous attempt raised the FilterBar inline tag cap from 6 to 10 and added a unit test. Find out why the cap is not taking effect in the running app (wrong component, wrong code path, cap bypassed, or different "header" surface than the one previously edited), then ensure the GUI header shows at most 10 tag chips in their existing order and collapses the remainder into a compact overflow indicator (e.g. `+N`). The underlying tag data and any tag displays outside the header must remain unchanged.

## Acceptance Criteria

- [x] Reproduced in the running GUI: identify the exact header component(s) that render more than 10 tag chips today and note the file/path in the Log.
- [x] In the running GUI, a context with more than 10 tags renders exactly 10 tag chips in the header tag area, in the existing order, plus a compact overflow indicator showing the count of hidden tags (e.g. `+3`).
- [x] In the running GUI, a context with 10 or fewer tags renders unchanged and shows no overflow indicator.
- [x] The cap is display-only: full tag data is preserved across save/reload, and tag chip lists outside the header (e.g. task detail, filter popovers) are unaffected.
- [x] A frontend test exercises the >10-tags and ≤10-tags cases against the component that actually drives the header, and fails if the cap is removed.
- [x] Manual verification step is recorded in the Log with the tag count used and the observed chip count + overflow text.
