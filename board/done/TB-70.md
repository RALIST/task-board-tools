# TB-70: BoardService.Triage(): wrap tb triage --json, cache + invalidate on watcher events

**Type:** feature
**Priority:** P2
**Size:** M
**Module:** gui
**Tags:** milestone-m6,wails-service,triage
**Branch:** —
**Parent:** TB-6

## Goal

Expose `BoardService.Triage(ctx) (map[string][]string, error)` to the frontend — keyed by task ID, value is the slice of reasons from `tb triage --json` (TB-69). The result is cached in memory; the watcher invalidates the cache on `board:reloaded` and `task:updated:<id>` (single-key invalidation) so the GUI doesn't re-shell `tb` per card render.

## Context

The card indicator (TB-73) needs an O(1) per-card "needs grooming?" check at render time. Shelling out to `tb` per card is unacceptable. The cache lives next to `BoardService.LoadBoard`'s in-memory snapshot and follows the same single-process invariant (one GUI per board). The watcher already emits the two event names the cache needs (`board:reloaded` for atomic CLI renames, `task:updated:<id>` for in-place edits — see TB-58). Invalidation strategy: `board:reloaded` clears the entire map; `task:updated:<id>` re-runs `tb triage --json` and updates that single entry (which may add or remove it).

## Acceptance Criteria

- [ ] `BoardService.Triage(ctx) (map[string][]string, error)` exists and is exported to Wails bindings; empty map (not nil) when no tasks need grooming
- [ ] First call shells `tb triage --json` and populates the in-memory cache; subsequent calls return the cached map until invalidated
- [ ] On `board:reloaded` from the watcher, the cache is cleared (lazy re-fetch on next `Triage()` call)
- [ ] On `task:updated:<id>`, the entry for that id is re-evaluated by re-running `tb triage --json` and updating that key (add / remove / replace) so cards reflect the change without a full reload
- [ ] Concurrency: cache reads + writes are guarded by a `sync.RWMutex`; tested with the race detector
- [ ] Integration test using a real `tb` binary: create a backlog task with placeholder Goal → `Triage()` returns the task with `["no goal"]`; `tb edit` filling the goal → `Triage()` no longer returns it
- [ ] The CLI command is invoked via the existing `cli.Exec` wrapper (TB-18); errors propagate as `error`, not panics
- [ ] No call site outside this service shells `tb triage` directly
