# TB-14: Implement active/archive/all status semantics

**Type:** improvement
**Priority:** P1
**Size:** S
**Module:** cli
**Tags:** milestone-m1,filters
**Branch:** —
**Parent:** TB-1

## Goal

Make tb status filters predictable: backlog, in-progress, done = concrete dirs; active = backlog+in-progress+done; archive = only archive; all = active + archive. Extend resolveStatus in cli/board.go and the collect helpers to read archive/. The current behaviour where --status all excludes archive is counter-intuitive.

## Context

Current semantics: `--status all` excludes archive, which is counter-intuitive. New canonical mapping in M1: `backlog`, `in-progress`, `done` = concrete dirs; `active` = `backlog+in-progress+done`; `archive` = only archive; `all` = active + archive. Old aliases `b`/`ip`/`d` still work. Update collect helpers to read `board/archive/` when implied.

## Acceptance Criteria

- [ ] `resolveStatus` in `cli/board.go` accepts `backlog`, `in-progress`, `done`, `active`, `archive`, `all`
- [ ] Aliases preserved: `b=backlog`, `ip=in-progress`, `d=done`
- [ ] Collect helpers read `board/archive/` when status implies it (archive or all)
- [ ] `tb ls --status archive` lists only archived tasks
- [ ] `tb ls --status all` lists active + archived
- [ ] `tb ls --status active` matches the previous `--status all` behaviour
- [ ] CLAUDE.md and docs/ARCHITECTURE.md updated with the new canonical mapping
- [ ] Help text (`cli/main.go` usage) reflects the new semantics
