## Goal

Task drawer must show task attachments and let users add, open, and remove them while the GUI stays a consumer of `tb` attachment commands instead of owning filesystem mutation logic.

## Context

- Parent epic **TB-93** defines the folder-task / attachments migration and the GUI acceptance requirement for picker add, remove, and open-in-OS.
- @gui/frontend/src/lib/components/TaskDrawer.svelte owns the drawer UI where the attachment list and controls should appear.
- @gui/frontend/src/lib/api.ts is the typed frontend wrapper surface for Wails calls from Svelte.
- @gui/app/ and @gui/internal/cli/ contain the GUI backend service and CLI wrapper surfaces that should invoke `tb`.
- **TB-99** owns `tb attach <ID> <path>...` and legacy file-task auto-promotion. **TB-100** owns `tb attach --rm <ID> <attachment-name>...`.

## Constraints

- Do not copy, move, write, or delete attachment files directly from the GUI process; add/remove must go through the GUI's `tb` wrapper.
- Do not reimplement CLI attachment validation, collision handling, or legacy auto-promotion in the GUI. Surface `tb` success/failure and refresh from the board after success.
- Keep drag-and-drop out of scope; **TB-104** owns drag/drop attachment entry points.
- Keep watcher/debounce architecture out of scope; **TB-105** owns watcher coalescing, while this task should consume one logical refresh after each drawer mutation.
- Follow the folder-task contract from **TB-94** for canonical paths and attachment metadata.

## Acceptance Criteria

- [x] TaskDrawer lists each attachment with file name, size, and an open-in-OS action.
- [x] Open-in-OS delegates to the OS shell through the GUI backend and reports a user-visible error if the OS refuses to open the file.
- [x] File-picker add supports selecting multiple files and calls `tb attach <ID> <path>...` via the GUI CLI wrapper; legacy file-task promotion is delegated to `tb`.
- [x] Per-row remove calls `tb attach --rm <ID> <attachment-name>` via the GUI CLI wrapper; the GUI does not delete files directly.
- [x] Successful add/remove refreshes the drawer from the board once and leaves the visible attachment list in sync without optimistic stale rows or flicker.
- [x] Failed add/open/remove surfaces a user-visible error and leaves the drawer state consistent with the board.

## Done when

## Manual test (UI)

Use a board with one folder-form task and one legacy file-form task. Open the folder task drawer; add 3 distinct files via the picker and verify rows show file names and sizes after the refresh. Click open-in-OS on one row and confirm the OS opens it. Remove one attachment and verify both the UI and `<status>/<ID>/attachments/` reflect the deletion. Repeat picker add and remove on the legacy file-form task and confirm `tb` auto-promotes it to folder form.
