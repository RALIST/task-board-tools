# TB-55: Daemon active-set dedup keyed by task_id

**Type:** feature
**Priority:** P1
**Size:** S
**Module:** gui
**Tags:** milestone-m5,agent,daemon,dedup
**Branch:** —
**Parent:** TB-5

## Goal

Keep an in-memory set of task IDs currently in the daemon's queue or in-flight, and refuse to re-enqueue a task that is already there. Prevents the watcher-fires-twice / startup-scan-races-watcher problem from spawning duplicate runs of the same task.

## Context

`AgentService.RunAgent` already has its own `activeRuns` map (TB-47) that prevents two manual runs of the same task. But that check happens inside `RunAgent` — by the time we get there, the daemon has already burned a semaphore slot and held the queue channel. The dedup belongs at the daemon's enqueue boundary: when the startup scan (TB-57) and the watcher subscription (TB-58) both observe the same task as `queued`, only the first attempt should enter the queue.

The active set tracks two states: `enqueued` (in the channel, not yet started) and `running` (worker has picked it up). Both are "in-flight" for dedup purposes. The entry is removed only after the worker releases the semaphore — i.e. terminal status reached.

This is *in-process* dedup — it does not survive restart. Cross-process correctness is the job of `.board.lock` and stale-running recovery (TB-60).

## Acceptance Criteria

- [ ] `Daemon` holds a `map[string]struct{}` guarded by a `sync.Mutex` (or `sync.Map`); accessor methods `tryEnqueue(id) bool` and `release(id)`
- [ ] `tryEnqueue` returns `false` and is a no-op when the ID is already present; returns `true` and inserts otherwise
- [ ] Workers call `release(id)` exactly once per task, in a `defer`, after the underlying run reaches a terminal status
- [ ] Cross-check with `AgentService` requires a new accessor `AgentService.HasActiveRun(taskID string) bool` (the existing `active map[string]*activeRun` is private at `gui/app/agent_service.go:66`). Daemon calls it inside `tryEnqueue` so a manual UI run that's still in-flight blocks a daemon enqueue for the same task_id.
- [ ] Unit test: feed the same task ID 100x concurrently → exactly one enqueue
- [ ] Unit test: enqueue, run-to-completion, re-enqueue the same task → second enqueue succeeds (slot released)
