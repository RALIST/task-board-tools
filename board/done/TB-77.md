# TB-77: Wire agent_timeout_minutes into agent_run.go (replace agentTimeoutDefault const)

**Type:** feature
**Priority:** P2
**Size:** S
**Module:** gui
**Tags:** milestone-m7,settings,agent
**Branch:** —
**Parent:** TB-7

## Goal

Replace the hardcoded `agentTimeoutDefault = 30 * time.Minute` in `gui/app/agent_run.go:21` with a per-run read of `SettingsService.GetAgentTimeoutMinutes()`, so changing the value via the M7 settings panel takes effect on the next run without an app restart.

## Context

Today every run gets a fixed 30-minute deadline (passed as `RunInput.Timeout` at `agent_run.go:369`). M7 makes this user-tunable via TB-76's `agent_timeout_minutes` preference. The read is per-run (not per-AgentService construction), so a value change in the settings panel propagates without restarting the daemon — distinct from `max_workers`, which is read once at daemon construction.

`AgentService` does not hold a SettingsService reference today. **Construction order matters**: `gui/main.go:41` builds `AgentService` *before* `SettingsService` is constructed at `gui/main.go:66`. The two options are:
- (a) inject a `TimeoutProvider func() time.Duration` into `AgentService` at construction, **late-bound** — pass a closure or a setter that the SettingsService is wired into after both exist (mirrors how `Emitter` is plumbed via `emitterShim` with a `getApp` getter to break the chicken-and-egg).
- (b) pass the SettingsService directly via a setter call after both are constructed.

Pick (a) — keeps `AgentService` ignorant of the settings package and matches the existing late-binding pattern. Until SettingsService is wired in, the provider must fall back to `AgentTimeoutMinutesDefault` so any pre-OpenBoard code path is safe.

## Acceptance Criteria

- [x] `agentTimeoutDefault` const removed from `agent_run.go`
- [x] `RunInput.Timeout` is computed from `GetAgentTimeoutMinutes() * time.Minute` at run-start time (not at AgentService construction)
- [x] `main.go` wires a late-bound `TimeoutProvider` (closure or setter) so `AgentService` reads live preferences per run; provider returns `AgentTimeoutMinutesDefault` when SettingsService is not yet bound
- [x] Test: bump preferences to `agent_timeout_minutes: 1`, run a long-running fake agent, observe `ErrTimeout` after ~1m (use faked clock or short-bound test if 1m is too slow); resetting the preference is honoured on the next run without restart
- [x] Run-log JSONL `started` event records the effective timeout (or a one-line note in code explaining why it doesn't)
