## Goal

Add typed Promise wrappers for the `SettingsService` preference getters/setters in `gui/frontend/src/lib/api.ts`, plus a `preferencesStore.ts` Svelte store that hydrates on mount and writes through on user changes — the data layer the M7 settings panel (TB-81) and default-agent dropdown (TB-79) depend on.

## Context

Existing pattern in `api.ts`: each Wails-generated `CancellablePromise` binding gets a thin `async function ...` wrapper. Stores live in `gui/frontend/src/lib/stores/` (see `runs.ts`, `selection.ts`, etc.). `preferencesStore.ts` should mirror `runs.ts` shape — `svelte/store` `writable` + `derived` (`runs.ts:14`), not Svelte 5 runes. The runes API is reserved for component state; the existing stores deliberately use the classic store API so `$store` autosubscriptions and `get(store)` work uniformly.

The store hydrates once at app start (call `loadPreferences()` from `+page.svelte` `onMount`). Set-methods optimistically update the store and roll back on backend error (toast on failure). No watcher integration — preferences.json is owned by the GUI process; nothing external mutates it.

## Acceptance Criteria

- [x] `api.ts` exports: `getMaxWorkers`, `setMaxWorkers`, `getAgentTimeoutMinutes`, `setAgentTimeoutMinutes`, `getDefaultAgent`, `setDefaultAgent`, `getCLIPath`, `setCLIPath` (all `Promise`-returning, mirroring existing wrapper style)
- [x] `gui/frontend/src/lib/stores/preferences.ts` (or `preferencesStore.ts`) exposes:
  - state: `{ maxWorkers, agentTimeoutMinutes, defaultAgent, cliPath, loaded }`
  - methods: `load()` (one-shot fetch), `set<Field>(value)` (optimistic write-through with rollback + toast on error)
- [x] Vitest unit tests cover: hydration sets `loaded=true`; set-method round-trips through the mocked API; rollback on rejected promise
- [x] No UI changes in this task — just the data layer
