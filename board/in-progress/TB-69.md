## Goal

Add `--json` to `tb triage` so the GUI can consume the grooming-needed list programmatically. JSON shape: `[{"id": "TB-12", "title": "...", "priority": "P2", "module": "core", "reasons": ["no module", "no goal"]}, ...]`. Empty result is `[]`, not prose — matches M1 JSON conventions (`docs/IMPLEMENTATION.md`).

## Context

`cli/triage.go` already collects `[]triageReason` (task + reasons) before pretty-printing. The JSON path just serialises that slice. The pretty path stays unchanged (M6 frontend uses JSON; humans keep the tabwriter view). Wire flag parsing via `flag.NewFlagSet` to stay consistent with the M1 ergonomics fix (TB-15). The CLAUDE.md JSON convention applies: stdout = data; stderr = errors/warnings; empty selection → `[]`, never "No tasks need grooming.".

## Acceptance Criteria

- [ ] `tb triage --json` prints a JSON array of `{id, title, priority, type, size, module, tags, status, filePath, reasons}` — camelCase keys matching `cli/json_output.go.marshalTask`
- [ ] `reasons` is a `[]string` from `checkNeedsGrooming` ("no priority", "no module", "no goal", "no acceptance criteria", "auto-created by scan")
- [ ] Empty result → stdout is exactly `[]\n`; no "No tasks need grooming." line
- [ ] Non-JSON path output is unchanged (test the existing tabwriter format)
- [ ] Flag is parsed via `flag.NewFlagSet` with `reorderArgs` for consistency with `tb show`
- [ ] `cli/board_test.go` (or a new `triage_test.go`) covers: empty board → `[]`, one task with two reasons → correct JSON, JSON validates as well-formed
- [ ] `cli/templates.go` and `cli/main.go` help blurbs reference `--json` for `tb triage`
