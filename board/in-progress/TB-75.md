# TB-75: CLI: tb edit --goal / --acceptance — body-section writes via writeFileAtomic

**Type:** feature
**Priority:** P2
**Size:** M
**Module:** cli
**Tags:** milestone-m6,cli,edit,body,groom
**Branch:** —
**Parent:** TB-6

## Goal

Extend `tb edit` so a groom agent can rewrite the `## Goal` and `## Acceptance Criteria` sections of a task body. Today `cli/edit.go` only mutates metadata fields (priority, type, size, module, tags, agent, agent-status). Without a body-edit surface the M6 groom prompt cannot fulfil F6.1 ("after the agent finishes, Goal section is improved on disk") — the agent has no allowed mutation that reaches the body.

## Context

Two CLI shapes considered:

- **Narrow flags** — `tb edit <ID> --goal <file|-> --acceptance <file|->` (one flag per supported section). Tightly matches the M6 scope (only Goal + AC). Body never grows additional sections via this command, so the surface stays predictable.
- **Generic flag** — `tb edit <ID> --section "## Goal" --content <file|->` (any heading). More flexible but lets an agent rewrite `## Log` or invent new sections — undesirable for a "groom" prompt that should not change history.

Pick the narrow path. The CLI semantics:

1. Reuses the existing `.board.lock` exclusive file lock (`lockBoard` in `cli/board.go`).
2. Reuses `writeFileAtomic` (the only sanctioned write path for task `.md` — the M1 invariant survives).
3. Replaces the body between `## Goal\n` and the next `## ` heading (similarly for `## Acceptance Criteria`). If the section is missing, insert it at a fixed position relative to existing sections (Goal after metadata block; Acceptance Criteria before `## Related Tasks` / `## Log`).
4. Appends an existing-style log entry: `- YYYY-MM-DD: Edited goal` / `acceptance`.
5. Calls `regenerateBoard` so `BOARD.md` stays in sync — `tb edit` already does this for metadata.
6. Read content from a file path or from stdin (`-`). No content allowed as a positional or inline flag value — files/stdin only, so multi-line bodies don't fight the shell.

The narrow flags coexist with the existing metadata flags; combining them in one invocation is allowed (e.g., `tb edit X -p P1 --goal goal.md`). Validation: empty content rejected; section name normalisation never strays from the two supported headings.

## Acceptance Criteria

- [ ] `tb edit <ID> --goal <file|->` replaces the content under `## Goal` with the file/stdin bytes (trimmed of leading/trailing blank lines) and leaves all other sections byte-identical
- [ ] `tb edit <ID> --acceptance <file|->` does the same for `## Acceptance Criteria`
- [ ] Both flags accept `-` (stdin) so a groom agent can pipe a heredoc without a temp file
- [ ] Missing section is inserted at a fixed, documented position (Goal after metadata; Acceptance Criteria before `## Related Tasks` / `## Log`, fallback to before `## Log` if no Related Tasks); existing sections retain anchor lines
- [ ] Empty content (after trim) is rejected with a non-zero exit and a clear message
- [ ] `## Log` and any other body section are untouched even if the content the agent supplied accidentally contains `## ` lines (the replacer stops at the next heading boundary, not at any `## ` substring)
- [ ] Operation goes through `lockBoard` + `writeFileAtomic`; `regenerateBoard` runs at the end
- [ ] Log entry appended: `- YYYY-MM-DD: Edited goal` or `Edited acceptance` (matches the existing edit-log style)
- [ ] Mixing metadata flags with `--goal`/`--acceptance` in one invocation works and appends a single combined log entry (e.g., `Edited priority=P1, goal`)
- [ ] New `cli/edit_body_test.go` (or extension of `cli/board_test.go`) covers: replace existing section, insert missing section, reject empty content, body roundtrip preserves trailing newline, `## ` lines in agent-supplied content do not split the section
- [ ] `cli/templates.go` and `cli/main.go` help text reflect the new flags
- [ ] The existing GUI `BoardService.EditTaskBody` direct-write path (TB-33) is unaffected — the GUI keeps its own surface; this task adds the CLI peer that the groom agent uses
