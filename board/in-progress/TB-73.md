# TB-73: Card: 'needs grooming' indicator + click-to-suggest-Groom in drawer

**Type:** feature
**Priority:** P2
**Size:** M
**Module:** gui
**Tags:** milestone-m6,frontend,card,triage,groom
**Branch:** —
**Parent:** TB-6

## Goal

`Card.svelte` shows a "needs grooming" indicator (icon + tooltip) when `triageStore.needsGrooming(task.id)` is true. The tooltip lists the reasons (`no module`, `no goal`, …). Clicking the indicator opens the drawer for that task and the Groom button (TB-72) is visually emphasised.

## Context

Cards live in `Column.svelte` and already subscribe to board state. Adding the indicator means reading from `triageStore` (TB-71) inside Card. The visual treatment should match the existing badge style (agent badge — TB-50) to keep the card density readable. Clicking the indicator must NOT trigger the card's own open-drawer handler twice — use `stopPropagation` on the indicator click and open the drawer explicitly, then publish a one-shot "highlight Groom" intent that TB-72's Groom button reads.

## Acceptance Criteria

- [ ] `Card.svelte` renders a "needs grooming" indicator when `$triageStore.needsGrooming(task.id)` is true
- [ ] Tooltip / title attribute shows the list of reasons joined with commas (matches the CLI's tabwriter format)
- [ ] Clicking the indicator opens the drawer for that card AND emits a one-shot `groom-suggest` signal that the drawer reads to apply the visual emphasis from TB-72's last AC (e.g., a writable store `groomSuggestedFor` set then cleared after one drawer-open cycle)
- [ ] The indicator does not appear on archive / done columns (only `backlog` — matches `tb triage`'s own scope)
- [ ] No layout shift when the indicator appears/disappears (reserve the space OR use `position: absolute` over the existing chrome)
- [ ] Vitest covers: indicator present / absent based on store state, tooltip text correct, click opens drawer, `groom-suggest` signal fires exactly once
