## Goal

Author `gui/internal/agent/prompts/groom.md` and expose it as the embedded `agent.PromptGroom` constant. The prompt instructs the agent to refine `## Goal` and `## Acceptance Criteria` via the narrow body-edit CLI surface added in TB-75 (`tb edit --goal -` / `tb edit --acceptance -`) and explicitly forbids code changes, edits to other body sections (Log, Related Tasks, Context), and moving the task off backlog.

## Context

M6 reuses the M4 runner plumbing — the only thing that changes for a "groom" run is the prompt and (in TB-66) a `Mode=groom` tag. The implement template (`prompts/implement.md`) is the style reference: short working contract, locked-down `{{TASK_ID}}/{{TASK_TITLE}}/{{TASK_BODY}}` placeholders rendered through `agent.RenderPrompt`. Adding a new placeholder is intentionally a three-file change (template + `PromptVars` + `RenderPrompt`) so reviewers see the full surface in one diff — keep the new template to the existing three tokens unless TB-66 needs more.

The prompt depends on TB-75 landing a real body-edit surface; today `cli/edit.go` only mutates metadata fields, so without TB-75 a groom agent has no allowed way to satisfy F6.1's "Goal section is improved on disk" acceptance. The prompt must instruct the agent to use `tb edit <ID> --goal -` / `--acceptance -` (piping a heredoc via stdin) — not free-form file edits to the task `.md`, which would bypass the board lock and atomic-write invariants.

## Acceptance Criteria

- [ ] `gui/internal/agent/prompts/groom.md` exists and uses only the three locked placeholders (`{{TASK_ID}}`, `{{TASK_TITLE}}`, `{{TASK_BODY}}`)
- [ ] `agent.PromptGroom` is `//go:embed prompts/groom.md`'d alongside `agent.PromptImplement`
- [ ] Prompt body tells the agent: (a) the only allowed mutations are `tb show {{TASK_ID}}` (read) and `tb edit {{TASK_ID}} --goal -` / `tb edit {{TASK_ID}} --acceptance -` (write via stdin heredoc) — see TB-75 for the surface, (b) do not change code or write files anywhere (including direct edits to the task `.md`), (c) do not edit `## Log`, `## Related Tasks`, `## Context`, or any other section, (d) do not move the task off backlog (no `tb start` / `tb done` / `tb mv`), (e) end with the task back in backlog with refined Goal + Acceptance Criteria
- [ ] Prompt includes a concrete example invocation showing the heredoc + stdin pattern so the agent does not invent a `--goal=<inline>` syntax
- [ ] `agent.RenderPrompt(PromptGroom, ...)` round-trips a non-empty body with all three placeholders substituted — covered by a table-driven test alongside the existing implement-prompt test
- [ ] No additional placeholders introduced (keeps `PromptVars` / `RenderPrompt` surface stable)
