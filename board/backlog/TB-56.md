# TB-56: Settings field max_workers (1-4) wired to daemon semaphore

**Type:** feature
**Priority:** P1
**Size:** S
**Module:** gui
**Tags:** milestone-m5,agent,daemon,settings
**Branch:** —
**Parent:** TB-5

## Goal

Expose `max_workers` as a persisted setting (default 1, clamped to `[1, 4]`) and use it as the daemon's semaphore capacity at startup. No UI in M5 — settings UI is M7 F7.1; M5 just lays the storage + wiring so the value can be configured by hand or by a future settings panel.

## Context

F5.3 specifies "Configurable via settings (1–4)". `SettingsService` already persists recent boards under `$XDG_CONFIG_HOME/tb-gui/recent.json` — but that file is a `[]RecentBoard` array, not a preferences object, and reshaping it would force a migration on every existing user. Add a separate `$XDG_CONFIG_HOME/tb-gui/preferences.json` with a `MaxWorkers int` field, validated on read.

M5 reads this value when constructing the daemon. Changing the value after the daemon is running requires a restart (acceptable for MVP — the daemon's lifecycle is the GUI session). Out of scope: hot-reload of `max_workers`.

The clamp matters: a `max_workers` higher than 4 can swamp the user's machine if the agents are CPU/IO heavy; lower than 1 makes no sense. Clamp on read, log a warning when the persisted value was out of range.

## Acceptance Criteria

- [ ] `SettingsService` (or a new `gui/app/preferences.go`) exposes `GetMaxWorkers() int` and `SetMaxWorkers(n int) error`
- [ ] Persistence: on-disk format is JSON in a stable file path (e.g. `$XDG_CONFIG_HOME/tb-gui/preferences.json`); default of 1 written if file missing or field absent
- [ ] Read-time clamp: values outside `[1, 4]` are coerced to the nearest bound, with a `slog.Warn` log entry
- [ ] `daemon.New` reads `MaxWorkers` from settings (via the Options struct populated in `main.go`) at startup; no hot-reload
- [ ] Unit test: round-trip write + read; clamping (`0` → 1, `999` → 4); missing file → default
- [ ] No frontend changes (M7 owns the UI)
