# TB-118: tb worktree adopt/list/status + cancel/fail preservation

**Type:** feature
**Priority:** P2
**Size:** M
**Module:** cli
**Branch:** —
**Parent:** TB-109

## Goal

Secondary worktree surface: migration adoption (`tb worktree adopt`), inspection (`list`, `status`), and the cancel/fail flow guarantees that worktrees survive interrupted runs for inspection.

The core removal command (`tb worktree clean`) was split into TB-122 because TB-113's branch-collision recovery and AC11 depend on it; this bundle stays P2.

## Context

Spec sections: § 4.9 (cancel), § 4.10 (failure), § 8 (migration via adopt), § 5 (CLI surface — `tb worktree list/status`).

## Acceptance Criteria

- [ ] `tb worktree adopt <ID> [--branch <existing-branch>]` for retrofit: takes an in-progress task, creates the worktree at `<worktrees.path>/<ID>` from the given branch (or `task/<ID>` if it already exists), writes a sidecar with `state: "active"`, `phase: "active"`. Idempotent — re-adopting an already-adopted task is a no-op.
- [ ] `tb worktree list` shows all active worktrees: ID, path, branch, base SHA, current phase. Reads sidecars + cross-checks `git worktree list`. Supports `--json`.
- [ ] `tb worktree status <ID>` shows worktree + git state for a single task (HEAD SHA, rebase-in-flight flag, clean/dirty, branch ahead/behind merge_target).
- [ ] **Cancel flow** (F4.4): when an agent is cancelled mid-run, the worktree is **not** auto-removed. Sidecar state stays as it was at cancel time. AgentStatus becomes `cancelled` per existing F4.4. User must `tb worktree clean <ID>` (TB-122) explicitly to remove.
- [ ] **Failure flow**: when an agent ends with `AgentStatus: failed`, the daemon does **not** attempt to merge. Worktree stays. User can retry the agent (the branch already has partial commits), abandon (`tb worktree clean`), or finalize manually.
- [ ] `tb mv <ID> backlog` (move back from in-progress) does **not** auto-remove the worktree — user must `tb worktree clean` explicitly.
- [ ] AC7 of the spec: cancel preserves worktree (verified by inspecting `<worktrees.path>/<ID>/` + sidecar state after a cancelled run).
- [ ] Migration test for `adopt`: create a board in pre-mode style (manually move task to in-progress, manually create branch), then enable worktree mode + `tb worktree adopt <ID> --branch <existing>` → next `tb done` runs the full TB-115 flow.
