# TB-116: tb merge: conflict retry path

**Type:** feature
**Priority:** P1
**Size:** M
**Module:** cli
**Branch:** —
**Parent:** TB-109

## Goal

New `tb merge <ID>` command. Finalizes a task whose previous `tb done` left a conflict (rebase mid-flight) or whose merge was otherwise interrupted. Always re-runs `git rebase` even after the user did `git rebase --continue` manually — target may have advanced during the conflict resolution window.

## Context

Spec sections: § 4.8, § 11 decisions 11, 17, 18.

## Acceptance Criteria

- [ ] New CLI command `tb merge <ID>`.
- [ ] Algorithm follows § 4.8:
  1. Take `.worktree.lock(<ID>)`.
  2. Read sidecar; validate `state: "active"`.
  3. Probe both `git rev-parse --git-path rebase-merge` AND `--git-path rebase-apply`. If either dir exists → fail with `rebase_in_progress` instructing the user to finish/abort first.
  4. Worktree clean? If not → `dirty_worktree`.
  5. From here: identical to `tb done` from step 4 (TB-115). Always re-run `git rebase <merge_target>` (no-op if already up to date; surfaces fresh conflicts if target advanced).
- [ ] Idempotent: running `tb merge <ID>` on an already-clean rebased branch finalizes successfully (no conflict was needed).
- [ ] AC4 of the spec: conflict scenario produces `MergeStatus: conflict` and leaves the rebase in-flight (`rebase-merge` or `rebase-apply` dir exists).
- [ ] AC5 of the spec: user resolves manually (`git rebase --continue`), runs `tb merge`, task finalizes to `done/`, MergeStatus cleared.
- [ ] AC8f of the spec: target advances during manual conflict resolution → `tb merge` re-runs rebase, surfaces the new conflict, loops cleanly.
- [ ] AC8h of the spec: forced `rebase-apply` backend is still detected by the probes.
