# TB-54: Worker pool calling an internal blocking executor on AgentService

**Type:** feature
**Priority:** P1
**Size:** M
**Module:** gui
**Tags:** milestone-m5,agent,daemon,pool
**Branch:** —
**Parent:** TB-5

## Goal

Inside the daemon, run N worker goroutines (N = `MaxWorkers`, default 1) that pull task IDs off an in-process channel and invoke a **new internal AgentService entry point that accepts an `AgentStatus=queued` task and blocks until terminal status**, plumbing the daemon's root context into the runner so `Daemon.Close()` cancels in-flight runs. The public `RunAgent` path is unchanged for the M4 drawer Run button.

## Context

Three real constraints, all verifiable in the M4 code:

1. **`RunAgent` rejects `queued`/`running`.** `gui/app/agent_run.go:98-100` short-circuits with `ErrAlreadyRunning` when `AgentStatus ∈ {queued, running}`. The daemon picks up tasks that are *exactly* in that state, so calling the public method is a non-starter.
2. **`RunAgent` returns before the run finishes** (`agent_run.go:161`'s `go s.runGoroutine(...)`), so its return cannot be the worker's release signal.
3. **The runner ctx is `context.Background()`** (`agent_run.go:148`), so daemon ctx cancellation does not reach the child process. Without a fix, `Daemon.Close()` cannot stop in-flight runs and TB-62's 5s grace becomes a 5s no-op.

The right shape is to **split public API from execution**:
- Public `AgentService.RunAgent` (drawer "Run" button) keeps its M4 contract: writes `queued` JSONL, sets `AgentStatus=queued`, registers `activeRun`, returns `run_id`. It no longer spawns directly; instead it hands off via the daemon's queue.
- New **internal blocking executor** — `AgentService.executeQueuedRun(ctx, id) (status string, err error)` (lowercase, unexported, daemon-only seam via an exported package method that takes a `daemonToken` or a friend-package method) — is what daemon workers call. It: validates `AgentStatus=queued`, writes `started`+pid+`agent` (TB-60 needs `agent` in `started`), sets `AgentStatus=running`, spawns runner with the passed-in ctx, runs the existing `runGoroutine` loop synchronously, writes `finished`+`AgentStatus terminal`, returns the status. Blocks until terminal; respects ctx cancel.
- The drawer's "Run" flow is now: drawer → public RunAgent (writes queued + emits event) → daemon's watcher subscription (TB-58) sees `AgentStatus=queued` → daemon worker calls `executeQueuedRun`. No code duplication, single spawn path.

Codex also flagged that the current `runGoroutine` does its JSONL+Wails+`tb edit` work while still holding `AgentService.mu` (acquired in `RunAgent` at `agent_run.go:106`). Daemon workers running in parallel would serialize behind that — and behind any in-flight `CancelRun`. The mutex must be narrowed to guard only `active` map insert/delete; JSONL/emit/tb-edit move outside.

Worker primitive: pick **N workers reading a buffered channel**, drop the redundant semaphore wording. That alone bounds parallelism to N.

## Acceptance Criteria

- [ ] `AgentService` exposes a daemon-only blocking executor (e.g. `RunQueuedAgentSync(ctx, id) (status string, err error)`) that:
  - accepts an `AgentStatus=queued` task without rejecting it (mirror inverse of the `agent_run.go:98-100` switch)
  - writes `started` JSONL with `pid` AND `agent` (TB-60 will read `agent` for the command-name cross-check; coordinate with TB-60 to update the event schema)
  - sets `AgentStatus=running` via `tb edit`, spawns the runner with the **caller-supplied ctx**, runs the existing `runGoroutine` loop synchronously, writes `finished` + terminal `AgentStatus`
  - blocks until terminal; returns `("success"|"failed"|"cancelled", err)`
- [ ] `AgentService.mu` is narrowed to guard only `active` map insert/delete and the `wasCancelled()` race; JSONL append, Wails emit, and `tb edit` happen outside the mutex
- [ ] `Daemon` owns a buffered `chan string` (task IDs); `N = MaxWorkers` worker goroutines loop `for id := range queue { exec(); release }` — no separate semaphore
- [ ] Workers pass the daemon's root ctx into the executor so `Daemon.Close()` cancellation propagates to `exec.CommandContext`
- [ ] Workers call `Daemon.release(id)` (TB-55) exactly once per task, in a `defer`, after the executor returns
- [ ] Workers exit cleanly when the queue channel is closed AND the daemon ctx is done (no goroutine leak; `goleak.VerifyNone(t)` in tests)
- [ ] **F5.3 acceptance**: with `MaxWorkers=1`, queue 3 tasks at once; they run strictly sequentially — verified by an integration test with an instrumented runner asserting `start[i+1] > finish[i]` on JSONL timestamps
- [ ] `go test ./gui/internal/daemon/...` covers: capacity 1 serializes; capacity 2 overlaps; ctx cancellation drains workers within 5s; closed queue drains gracefully
