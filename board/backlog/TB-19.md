# TB-19: BoardService: LoadBoard and GetTask via exec tb

**Type:** feature
**Priority:** P1
**Size:** M
**Module:** gui
**Tags:** milestone-m2,wails-service
**Branch:** —
**Parent:** TB-2

## Goal

Implement gui/app/board_service.go with LoadBoard() (exec tb ls --json --status active) and GetTask(id) (exec tb show <id> --json). Exposed as Wails bindings. Returns typed structs that the frontend can consume directly. No mutations in M2 — those land in M3.

## Context

First Wails service exposed to the frontend. Read-only by design — every mutation in M2 stays in the CLI's lane. The board snapshot returned must match the JSON shape that TB-12 defines so the frontend can use it directly without an adapter layer.

## Acceptance Criteria

- [ ] `gui/app/board_service.go` declares `BoardService` with `LoadBoard(ctx) (BoardSnapshot, error)` and `GetTask(ctx, id string) (TaskDetail, error)`
- [ ] `LoadBoard` shells out via `cli.Client.RunJSON(... "ls", "--json", "--status", "active")` and returns the parsed snapshot
- [ ] `GetTask` shells out via `... "show", id, "--json"` and returns `{metadata, body}`
- [ ] Service registered with Wails so it appears in generated frontend bindings
- [ ] `BoardSnapshot` shape mirrors the CLI JSON contract (TB-12); types annotated with TS-friendly JSON tags
- [ ] Empty board → `BoardSnapshot` with empty arrays, no error
- [ ] Unknown ID for `GetTask` → typed `ErrNotFound`, surfaced as a Wails error the frontend can handle
