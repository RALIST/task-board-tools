# TB-117: Worktree recovery on startup

**Type:** feature
**Priority:** P1
**Size:** M
**Module:** cli
**Branch:** —
**Parent:** TB-109

## Goal

Extend existing M5 stale-recovery to reconcile crashes in the worktree lifecycle. Two scans (sidecar scan + orphan worktree scan), then a phase-based reconciliation table that respects user intervention (taskDir-aware `finalizing` row, no auto-correction of user moves).

## Context

Spec sections: § 4.12 (the full table), § 4.13 (corrupt sidecar handling), § 11 decisions 12, 16, 20, 23.

The reconciliation table is the linchpin — every phase has a defined recovery, and crashes between any two phase writes resolve to a known good state.

## Acceptance Criteria

- [ ] **Scan A — sidecar scan.** For each `<board>/.worktrees/<ID>.json`, run the § 4.12 reconciliation table. Do **not** short-circuit on "worktree path missing" — the table is the single source of truth.
- [ ] **Scan B — orphan worktree scan.** For each entry in `git worktree list` under `<worktrees.path>/<ID>` with no matching sidecar → log to `tb doctor`. Do not auto-delete.
- [ ] Recovery helpers implemented: `wtExists`, `branchExists`, `taskDir`, `mergeContainsBranch` (using `git merge-base --is-ancestor` — ancestry, not equality), `rebaseInFlight` (probes both `rebase-merge` and `rebase-apply`).
- [ ] Reconciliation table covers all rows in § 4.12:
  - `creating`/(any) → cleanup all artifacts.
  - `active`/`active`/`backlog` → idempotently finish the `tb start` file move.
  - `active`/`active`/`in-progress` → no-op (steady state).
  - `active`/`active`/`done|archive|missing` → log to `tb doctor`.
  - `active`/`rebasing`/(any) → probe `rebaseInFlight`; if yes set `MergeStatus: conflict`; if no reset phase to `active`.
  - `active`/`finalizing`/`in-progress` → if `mergeContainsBranch` finish file move + cleanup; else reset phase to `active`.
  - `active`/`finalizing`/`done` → if `mergeContainsBranch` proceed to cleanup; else log to `tb doctor`. Do not auto-move.
  - `active`/`finalizing`/`archive` → if `mergeContainsBranch` clean up worktree+branch+sidecar but **do not touch task file** (respect user archive); else log to `tb doctor`.
  - `active`/`finalizing`/`backlog|missing` → log to `tb doctor`.
  - `active`/`cleanup`/`done|archive` → idempotent retry of git/sidecar cleanup.
  - `active`/`cleanup`/other → log to `tb doctor`.
- [ ] Corrupt sidecar (unreadable JSON): log to `tb doctor`, skip entry, leave all artifacts (§ 4.13 rule).
- [ ] Recovery actions acquire locks in the standard order: `.worktree.lock(ID)` → `.board.lock` (when needed for file move). Never `.merge.lock` during recovery.
- [ ] AC8a of the spec: rebasing crash → conflict.
- [ ] AC8b of the spec: finalizing crash post-FF → idempotent finalize (not "conflict").
- [ ] AC8c of the spec: cleanup crash → idempotent cleanup.
- [ ] AC8d of the spec: backlog-task-with-active-worktree → file move completed.
- [ ] AC8e of the spec: ancestry-not-equality detection after subsequent merges advanced the target.
- [ ] AC8g of the spec: corrupt sidecar.
- [ ] AC8i of the spec: user archived the task during the recovery window → file untouched, worktree cleaned.
