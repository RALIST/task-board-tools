# TB-122: tb worktree clean: removal command

**Type:** feature
**Priority:** P1
**Size:** S

**Branch:** —

## Goal

The `tb worktree clean <ID> [--force]` command: remove worktree directory, delete the `task/<ID>` branch, delete the sidecar. Required for branch-collision recovery (TB-113) and for AC11. Carved out of TB-118 so the core flow doesn't depend on a P2 bundle.

## Context

Spec sections: § 4.5 step 3 ("user must explicitly `tb worktree clean <ID>` first"), § 4.11 (the command itself).

## Acceptance Criteria

- [ ] New CLI subcommand `tb worktree clean <ID> [--force]` registered in `cli/main.go`.
- [ ] Algorithm:
  1. Take `.worktree.lock(<ID>)`.
  2. Read sidecar if present; resolve worktree path + branch (or fall back to convention if sidecar absent and `--force`).
  3. Refuse without `--force` UNLESS:
     - The task is in `done/` or `archive/` (already finalized), or
     - The branch has no commits beyond the recorded `baseSHA` (nothing to lose).
  4. With permission to proceed: `git worktree remove <path>` (use `--force` to git only if `--force` was passed to `tb`), `git branch -d task/<ID>` (or `-D` with `--force`), delete sidecar atomically.
  5. Release lock.
- [ ] Reports what would be lost when refusing (commit count beyond `baseSHA`, current `phase` in sidecar, etc.).
- [ ] `--force` skips all gating checks but still releases locks correctly on failure.
- [ ] AC11 of the spec: refuses without `--force` when un-merged commits exist; succeeds with `--force`.
- [ ] Integration test: run after a failed `tb start` (orphan branch) → `tb worktree clean --force` cleans the orphan, allows a fresh `tb start <ID>` to succeed.
