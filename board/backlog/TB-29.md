# TB-29: parseTaskFile should reject malformed task files

**Type:** bug
**Priority:** P3
**Size:** S
**Module:** cli
**Tags:** robustness,json,parser,quick-win
**Branch:** —

## Goal

Prevent malformed task files from surfacing as empty-ID / empty-title JSON objects by making `parseTaskFile` reject missing or malformed task headers and making each caller handle that parser error deliberately.

## Context

`cli/task.go:parseTaskFile` currently returns a zero-value `Task` and `nil` error if the first `maxMetadataLines` lines do not contain a valid `# <PREFIX>-<N>: <title>` header. That means:

- `tb ls --json` can emit objects with `id: ""` / `title: ""` because `cmdList` only skips when `parseTaskFile` returns an error.
- `tb board --json` and regenerated `BOARD.md` can silently include or compute from zero-value tasks because `collectTasks` has the same skip-on-error pattern.
- `tb grep` and `tb triage` also already skip parser errors, but today malformed task headers are not parser errors.
- `tb show <ID> --json` should fail loudly for the selected task instead of returning `{metadata:{id:"",title:""}, body:"..."}`.

The current GUI board path reads through `tb ls --json` (`BoardService.LoadBoardWithMode`), so the CLI JSON contract is the immediate protection surface. There is no separate `gui/internal/parser` package today; do not add one in this task.

## Scope Decision

Define a single malformed-task contract in the CLI parser:

- A task is malformed when no header matching the configured prefix is found in the first `maxMetadataLines` lines.
- A task is malformed when the header has an ID but no title separator/title, for example `# TB-123`, `# TB-123:`, or `# TB-123:    `.
- Keep metadata fields optional; do not reject a file for missing `Type`, `Priority`, `Size`, `Module`, `Tags`, `Branch`, `Parent`, `Agent`, or `AgentStatus`.
- Bulk views (`ls`, `board`, `grep`, `triage`) should warn to stderr with the path and skip the bad file. They should continue returning valid tasks and valid JSON on stdout.
- Single-task JSON (`show --json`) should exit non-zero with a clear parse error for the requested task.

## Acceptance Criteria

- [ ] `parseTaskFile` returns a non-nil error when the header is absent in the first `maxMetadataLines` lines.
- [ ] `parseTaskFile` returns a non-nil error when the header starts with `# <PREFIX>-<N>` but lacks a non-empty title after `:`.
- [ ] Valid tasks with optional metadata omitted still parse successfully; this task does not introduce required metadata beyond ID/title.
- [ ] `tb ls --json --status all` skips malformed files, emits a warning to stderr that includes the file path, and stdout remains valid JSON containing only valid tasks.
- [ ] `tb board --json` and `tb board` skip malformed files with the same warning pattern; `BOARD.md` regeneration never includes an empty-ID row.
- [ ] `tb grep <pattern>` and `tb triage` skip malformed files with a warning if they encounter a parser error.
- [ ] `tb show <ID> --json` exits non-zero with a clear parse error when the selected file is malformed.
- [ ] Tests cover parser unit cases plus command-level JSON behavior for at least `ls --json`, `board --json`, and `show --json`.
- [ ] `cd cli && go test ./...` passes.

## Non-Goals

- Do not validate every metadata enum or require metadata fields; existing lenient metadata parsing stays in place.
- Do not infer task identity from the filename when the markdown header is missing.
- Do not add or refactor a GUI parser package.
- Do not change atomic-write or watcher behavior.
