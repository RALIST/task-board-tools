# TB-68: Daemon: honor Mode=groom from queued JSONL event so pickup uses GroomingDecorator

**Type:** feature
**Priority:** P2
**Size:** S
**Module:** gui
**Tags:** milestone-m6,agent,daemon,groom
**Branch:** —
**Parent:** TB-6

## Goal

Teach `AgentService.RunQueuedAgentSync` (the daemon-only blocking executor) to read `Mode` from the queued JSONL event for the task and, when `Mode=groom`, wrap the factory-produced runner with `agent.GroomingDecorator` before passing it into the shared run path. This closes the loop for `tb edit X --agent-status queued` from the terminal on a task whose intended mode was groom: the daemon picks it up and uses the groom prompt — matching what the GUI's `GroomTask` (TB-67) would have done.

## Context

`RunQueuedAgentSync` already handles two queue sources (the GUI's `RunAgent` and the CLI's `tb edit --agent-status queued`) and synthesises a `queued` event when one is missing (see `findQueuedRunID` / `ErrNoQueuedRun`). Today it hard-codes `Mode=implement`. M6 adds a third queue source — `GroomTask` — and the daemon must respect the recorded mode. The cleanest fix is to make `findQueuedRunID` return the JSONL event's `Mode` alongside its `RunID`, and have the synthetic path default to `ModeImplement` (the CLI doesn't know about groom). When the resolved mode is `ModeGroom`, wrap the runner exactly as TB-67 does.

## Acceptance Criteria

- [ ] `findQueuedRunID` (or its replacement) returns `(runID string, mode agent.Mode, err error)` so the executor learns the mode from the JSONL event rather than re-deriving it
- [ ] `RunQueuedAgentSync` wraps the factory runner with `agent.NewGroomingDecorator(inner, agent.PromptVars{TaskID, TaskTitle, TaskBody})` iff the resolved mode is `ModeGroom`; vars sourced from the same `BoardService.GetTask` lookup the executor already performs
- [ ] `activeRun.Mode`, the synthetic `queued` event (when one is missing), the `started` event, the `finished` event, and the `agent:run-queued` Wails emit all carry the resolved mode (defaults to `implement` for the CLI-direct case)
- [ ] Daemon shutdown (`finishCancelled(reason: "shutdown")`) preserves the mode on the synthetic `finished{cancelled}` record
- [ ] Integration test: write a JSONL `queued` event with `mode=groom`, set `AgentStatus=queued`, kick the daemon — the run uses the groom prompt (asserted via stub Runner capturing `RunInput.Prompt`) and JSONL `finished` records `mode=groom`
- [ ] CLI-only path (no JSONL event, `tb edit --agent-status queued`) still defaults to `mode=implement` — covered by existing `RunQueuedAgentSync` tests with no regression
