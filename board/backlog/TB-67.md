# TB-67: AgentService.GroomTask: queue a groom run distinct from a normal run

**Type:** feature
**Priority:** P2
**Size:** M
**Module:** gui
**Tags:** milestone-m6,agent,wails-service,groom
**Branch:** —
**Parent:** TB-6

## Goal

Add `AgentService.GroomTask(ctx, id) (runID, error)` — an M4-compatible twin of `RunAgent` that records `Mode=groom` in the `activeRun`, in the JSONL `queued`/`started`/`finished` events, and in the Wails `agent:run-queued` payload. It wraps the factory-produced runner with `agent.GroomingDecorator` before handing it to the goroutine. Shares the `active` map, `postRun`, `finishCancelled`, and Cancel surface so a groom run cancels exactly the same way as an implement run.

## Context

`RunAgent` and the M5 daemon executor `RunQueuedAgentSync` are the two entry points into the run lifecycle. M6 must NOT fork them — the shared `runGoroutine` / `postRun` / `finishCancelled` paths are what make `AgentStatus` round-tripping crash-safe (M5 invariants). The wins of the decorator-in-AgentService approach: (1) the mode-aware swap lives at one call site, (2) JSONL `started` keeps `agent: "claude"|"codex"` for TB-60's `pidAlive` cross-check, (3) past-runs (TB-49 / TB-52) just learn a new `mode` field to render the label. Surface a single `GroomTask` Wails method so the frontend doesn't need a parallel cancel API — the existing `CancelRun(id)` cancels whichever run is active.

## Acceptance Criteria

- [ ] `AgentService.GroomTask(ctx context.Context, id string) (string, error)` exists and is exported to Wails bindings
- [ ] Step ordering matches `RunAgent` (TB-47): (1) state validation including `ErrAlreadyRunning` against the shared `active` map, (2) JSONL `queued` with `mode=groom`, (3) Wails `agent:run-queued{mode: "groom"}`, (4) `tb edit --agent-status queued`, (5) goroutine kick that runs the decorated runner
- [ ] `activeRun.Mode` is set to `agent.ModeGroom.String()` so cancel/shutdown paths know the mode
- [ ] The runner passed to `runGoroutine` is `agent.NewGroomingDecorator(factory(agentName), agent.PromptVars{TaskID, TaskTitle, TaskBody})` — vars sourced from the same `BoardService.GetTask` lookup that `RunAgent` already performs
- [ ] Because the decorator overwrites `RunInput.Prompt`, the existing `runGoroutine` `RenderPrompt(PromptImplement, …)` call site is allowed to stay as-is (its output is discarded by the decorator) — no fork of `runGoroutine` for groom
- [ ] JSONL `started` and `finished` events also carry `mode=groom` (uses the existing `Event.Mode` field that already exists in M4 `state.go`)
- [ ] An attempt to `GroomTask` a task that has no agent assigned returns `ErrNoAgent`; an attempt while a run (groom or implement) is in flight returns `ErrAlreadyRunning`
- [ ] Cancelling a groom run via the existing `CancelRun(id)` flows through the same 5-step ordering (mark → kill → JSONL `finished{cancelled, mode: groom}` → Wails `agent:run-finished` → `tb edit --agent-status cancelled`)
- [ ] Unit tests with the runner-factory stub: groom run succeeds, fails, cancels — JSONL has `mode=groom` on every event for that run_id; `AgentStatus` round-trips queued→running→success|failed|cancelled exactly like `RunAgent`
- [ ] `RunAgent` behaviour is byte-for-byte unchanged — covered by re-running the existing `agent_run_test.go` suite
