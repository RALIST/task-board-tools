# TB-111: Worktree config + preflight + gitignore

**Type:** feature
**Priority:** P1
**Size:** S
**Module:** cli
**Branch:** —
**Parent:** TB-109

## Goal

Foundation for worktree mode. Adds the `worktrees.*` keys to `.tb.yaml`, materializes the gitignore patterns, and implements the preflight refusals (submodules, path conflicts). Nothing in the runtime flow uses worktrees yet — this just makes the config and environment ready.

## Context

Spec sections: § 4.1 (layout + gitignore), § 4.5 preflight, § 11 decisions 8, 19, 22.

## Acceptance Criteria

- [ ] `.tb.yaml` accepts and persists the `worktrees` block:
  ```yaml
  worktrees:
    enabled: true|false       # default true on fresh `tb init`, false on existing boards
    path: .tb-worktrees       # default
    branch_prefix: task/      # default
    merge_target: main        # default
    fetch_before_merge: false # default
    allow_submodules: false   # default (override for submodule preflight)
  ```
- [ ] `tb init` on a fresh repo writes the block with defaults and adds gitignore patterns: `<worktrees.path>/`, `<board>/.worktrees/`, `<board>/.worktree-locks/`, `<board>/.merge.lock`. Patterns resolve from the configured board path.
- [ ] Enabling `worktrees.enabled: true` on an existing board (where the key was absent or `false`) runs preflight: refuses if `.gitmodules` exists (unless `allow_submodules: true`) and refuses if `<worktrees.path>/` exists and is non-empty / not previously managed by `tb`.
- [ ] Gitignore writes are idempotent: re-running `tb init` does not duplicate lines.
- [ ] AC12 of the spec: submodule preflight blocks `tb start` with a clear error; `allow_submodules: true` override lets it through.
- [ ] AC13 of the spec: path-conflict preflight blocks `tb start` with clear instructions.
