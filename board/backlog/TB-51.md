# TB-51: AgentRunLog.svelte: live streaming and past-run log rendering

**Branch:** —
**Parent:** TB-4

## Goal

The streaming-log pane inside the drawer. Two modes from one component: subscribe to live Wails events for an active run, or render the `.log` file content for a past run.

## Acceptance Criteria

- [ ] Accepts a `runID` prop. When the corresponding `Run` (from `runsStore`) has `status === 'running'`, subscribes to `agent:run-log` and appends each matching (`run_id` matches) event line to an internal buffer
- [ ] When the run is in any terminal status (`success | failed | cancelled`), fetches the log file content via `AgentService.GetRunLog(taskID, runID)` (from TB-49) and renders it as a static `<pre>` block. No reading directly from `Run.LogPath` on the frontend — always go through the service so path resolution stays backend-owned
- [ ] **F4.3 latency**: a `agent:run-log` event arriving at the Wails frontend appears in the rendered DOM within ~1 second. Verified by a Vitest test that emits an event and uses `vi.advanceTimersByTime`/`tick()` to assert the DOM update happens inside that budget (no `requestAnimationFrame` throttling, no debouncing of log lines)
- [ ] Sticky-bottom scroll: auto-scrolls to the newest line by default; if the user scrolls up, stops auto-scrolling until they scroll back to the bottom
- [ ] Long lines wrap; ANSI escape sequences stripped before render via a small `stripAnsi` helper (so colourised agent output doesn't bleed control codes into the DOM)
- [ ] Status pill (`queued | running | success | failed | cancelled`) at the top — sourced from the `Run` in `runsStore` (which TB-52 keeps in sync via Wails events)
- [ ] Tests (Vitest): mounting with a finished run renders its log content via `GetRunLog`; emitting 5 `agent:run-log` events for an active run appends 5 lines in order within ~1s; sticky-bottom respects the user's scroll position; `GetRunLog` returning `ErrNotFound` renders an empty pane instead of crashing
