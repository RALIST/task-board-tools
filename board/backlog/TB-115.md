# TB-115: tb done: three-case FF + merge-lock + phase transitions

**Type:** feature
**Priority:** P1
**Size:** L
**Module:** cli
**Branch:** —
**Parent:** TB-109

## Goal

Extend `tb done <ID>` to rebase the task branch onto `<merge_target>`, fast-forward it back into the target via the three-case algorithm (A: merge in main, B: ref-push, C: refuse external), move the task file under `.board.lock` revalidation, and clean up the worktree. Phase-tagged sidecar lets recovery (TB-117) reconcile every crash point.

## Context

Spec sections: § 4.7 (the algorithm), § 6 (lock ordering rules), § 4.13 (sidecar atomicity), § 11 decisions 9, 11, 15, 21, 22.

Biggest single piece of the epic. Most of the Codex review iterations were about this section.

## Acceptance Criteria

- [ ] New global merge-lock at `<board>/.merge.lock` (flock). Implements the lock-ordering invariant from § 6: `.worktree.lock(ID)` → `.merge.lock` → `.board.lock`. Never acquire `.merge.lock` while holding `.board.lock`.
- [ ] New per-task metadata field `**MergeStatus:** none | conflict` parsed by `parseTaskFile`, settable internally (not user-facing via `tb edit`).
- [ ] `tb done <ID>` algorithm follows § 4.7 step-by-step:
  1. Take `.worktree.lock(<ID>)`.
  2. Read sidecar, validate `state: "active"`, resolve paths.
  3. Worktree clean? Probe both `rebase-merge` and `rebase-apply` for in-flight rebase. Fail with `dirty_worktree` / `rebase_in_progress` if not.
  4. Take `.merge.lock`.
  5. Write sidecar `phase: "rebasing"` atomically.
  6. Run `git rebase <merge_target>` in worktree. On conflict: **do NOT abort** — leave mid-flight; take `.board.lock`, set `MergeStatus: conflict`, append Log, regenerate; release locks; return error.
  7. Rebase clean: write sidecar `phase: "finalizing"`.
  8. Take `.board.lock`. Re-validate task state (file at `in-progress/<ID>.md`, branch field matches if set). If drifted → `state_drift` error.
  9. Three-case FF-finalize (still under `.board.lock`):
     - **Case A** — `mainHead == <merge_target>` AND externals is empty: `git -C <main> merge --ff-only task/<ID>`. No blanket `git status --porcelain` pre-check (board/ is dirty by design; the task branch's skip-worktree on `<board>/` means the FF diff never touches it).
     - **Case B** — externals is empty (and `mainHead != <merge_target>`): `git -C <main> push . task/<ID>:<merge_target>`.
     - **Case C** — externals is non-empty (any external worktree has `<merge_target>` checked out, excluding our own `<worktrees.path>/<ID>`): refuse with `merge_target_checked_out_externally`.
  10. Move task file `in-progress/<ID>.md` → `done/<ID>.md`, append Log "Done — merged at <SHA>", clear `MergeStatus`, regenerate `BOARD.md`. Release `.board.lock`.
  11. Write sidecar `phase: "cleanup"`.
  12. `git worktree remove <wt>`, `git branch -d task/<ID>`, delete sidecar.
  13. Release `.merge.lock` and `.worktree.lock`.
- [ ] All phase-transition sidecar writes use `writeFileAtomic`.
- [ ] **Degraded `tb done` path for pre-mode tasks** (§ 8 migration): when `worktrees.enabled: true` but the target task has no sidecar at `<board>/.worktrees/<ID>.json`, `tb done` skips steps 4–13 and runs only today's plain behavior — `in-progress/<ID>.md` → `done/<ID>.md`, append Log, regenerate `BOARD.md`. No rebase, no merge, no git operations. The user is responsible for any branch they may have created manually.
- [ ] AC3a–e of the spec: FF works in main-on-target case, ref-push in unrelated-branch case, refusal in external-worktree case, handles dirty board/ in main, propagates `merge --ff-only`'s refusal on conflict with user's other uncommitted code.
- [ ] AC10 of the spec: dirty worktree fails with `dirty_worktree`.
- [ ] Migration test: with `worktrees.enabled: true`, create a task file directly in `in-progress/<ID>.md` without going through `tb start` (so no sidecar at `<board>/.worktrees/<ID>.json`). Run `tb done <ID>` — expect exit success, task moved to `done/`, Log appended, no rebase or git operations performed.
