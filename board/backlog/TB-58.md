# TB-58: Watcher event sink: enqueue on task changes via emitter fan-out

**Type:** feature
**Priority:** P1
**Size:** M
**Module:** gui
**Tags:** milestone-m5,agent,daemon,watcher
**Branch:** —
**Parent:** TB-5

## Goal

Give the daemon an in-process feed of watcher events without changing `watcher.Watcher`'s public API. When the watcher reports a change (either `task:updated:<id>` for in-place writes or `board:reloaded` for atomic renames / column changes), the daemon re-checks tasks and `tryEnqueue`s any that are now `AgentStatus=queued` with an `Agent` set. This is the live-pickup path (CLI edit while GUI is open, automation, M6 groom).

## Context

Two facts about the current code, both verifiable:

1. **`board:reloaded` is fired for atomic CLI edits, not `task:updated:<id>`.** `watcher.handle` (`gui/internal/watcher/watcher.go:201-211`) dispatches by op: `Create | Remove | Rename → board:reloaded`; only bare `Write → task:updated:<id>`. The CLI's mandated atomic-write path is temp + fsync + `os.Rename`, which fsnotify reports as `Rename` (plus `Create` of the new inode). So `tb edit X --agent-status queued` produces a `board:reloaded`, not a `task:updated:<id>`. A daemon that only watched `task:updated:<id>` would miss every CLI-driven queue event — the exact scenario F5.1 calls out.

2. **`watcher.Watcher` already has the minimum-invasive seam: its `Emitter` interface** (`watcher.go:50-55`). `main.go:38-39` constructs an `emitterShim` and passes it in. The daemon does not need a new `Subscribe()` public API — it can register a second emitter (or a tee shim that wraps the existing one and also pushes to a daemon-side channel). No change to `watcher.Watcher`'s public surface.

Design:
- In `main.go`, replace `emitterShim` with a tee that forwards to both the Wails app event bus AND a daemon-side `chan watcherEvent`. Or expose a `daemon.Sink` that *implements* `watcher.Emitter` and chain it: `watcher.New(teeEmitter{wails, daemon}, ...)`. Either way, no edit to `watcher.go`.
- Daemon goroutine reads events of name `task:updated:*` AND `board:reloaded`:
  - `task:updated:<id>`: re-parse task `<id>` via `BoardService.GetTask`; if `AgentStatus=queued && Agent != ""`, `tryEnqueue`.
  - `board:reloaded`: run a quick re-scan via `BoardService.LoadBoard("active")` filtered to `AgentStatus=queued`, then `tryEnqueue` each. This catches the atomic-rename path. The scan is cheap (in-process, no shell-out) and the active-set dedup (TB-55) eliminates double-enqueues.
- The 200ms board-reload debounce already lives in the watcher; no extra debounce needed in the daemon.
- A "last-seen AgentStatus per task" cache is *not* required — `tryEnqueue` already dedups, and the per-event re-parse is cheap.

## Acceptance Criteria

- [ ] `watcher.Watcher`'s public API is unchanged. Daemon receives events via a second `watcher.Emitter` implementation (tee/fan-out) wired in `gui/main.go`
- [ ] Daemon registers the sink during construction; a goroutine reads events until daemon ctx is done
- [ ] On `task:updated:<id>`: `GetTask(id)`; if `AgentStatus=queued && Agent != ""`, `tryEnqueue`
- [ ] On `board:reloaded`: `LoadBoard("active")`, filter to `AgentStatus=queued && Agent != ""`, `tryEnqueue` each. **Rationale**: atomic CLI rename only fires `board:reloaded`, so this branch is load-bearing for F5.1.
- [ ] **F5.1 live acceptance**: with GUI running, terminal runs `tb edit X -a claude --agent-status queued` — within 5s task transitions to `running` and drawer shows live log output. The relevant fsnotify event observed by the watcher in this scenario is `Rename` → `board:reloaded` (not `Write` → `task:updated:<id>`); the test should assert the enqueue path through `board:reloaded`.
- [ ] Integration test: spawn watcher + daemon over tempdir board; do a CLI-style atomic write of a queued task; assert enqueue within 1s
- [ ] No regression in existing watcher tests; Wails event emission to the frontend unchanged
- [ ] Sink goroutine handles burst/back-pressure (buffered chan; drop with WARN log if full — recovery is fine because next watcher event re-triggers scan)
