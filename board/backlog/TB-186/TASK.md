# TB-186: Change parent task

## Goal

Ship task-page parent reassignment in the GUI by adding a safe CLI relationship mutation, exposing it through the GUI backend/API, and adding drawer UI for changing or clearing a task's parent epic.

## Context

- Today a task's parent is stored as `**Parent:** TB-NNN` metadata, while the parent epic also carries a generated `## Subtasks` bullet for the child.
- Creating a child with `tb create --parent` keeps those two sides in sync, but there is no equivalent safe path for changing a parent after the task exists.
- `gui/frontend/src/lib/components/TaskDrawer.svelte` is the task page/drawer where metadata is edited today; `gui/app/board_service.go` and `gui/internal/cli/mutations.go` route structured GUI mutations through `tb`.
- This epic is sibling UX to quick child creation and parent/child navigation tasks.

### Constraints

- Split implementation across the child tasks; do not ship parent editing by direct frontend or backend markdown rewrites.
- Preserve board invariants: structured mutations take `.board.lock`, task writes are atomic, parent and child files stay consistent, and `BOARD.md` is regenerated by the CLI path.
- Support clearing a parent as well as moving a child between two parent epics.
- Keep task status, archive semantics, attachments, and the existing markdown format out of scope.

## Subtasks

- **TB-191** (M) — CLI: safely reassign a task parent
- **TB-192** (S) — GUI backend: expose parent reassignment
- **TB-193** (M) — TaskDrawer: edit parent epic from task page
## Acceptance Criteria

- [ ] **TB-191** is done: the CLI can add, change, and clear a task's parent while keeping child `Parent` metadata and old/new parent `## Subtasks` sections consistent.
- [ ] **TB-192** is done: the GUI backend, internal CLI wrapper, generated bindings, and typed frontend API expose the parent reassignment mutation without direct markdown writes.
- [ ] **TB-193** is done: the task drawer lets a user change or clear parent epic from the task page and handles validation failures through the existing UI feedback path.
- [ ] End-to-end behavior is verified on a board with two epics and one child: move the child from old epic to new epic, confirm both epic bodies update, then clear the parent and confirm no stale parent/subtask data remains.
- [ ] Regression verification for the completed epic includes `cd cli && go test ./...`, `cd gui && go test ./...`, `cd gui/frontend && npm run check`, and `cd gui/frontend && npm test -- --run`.
