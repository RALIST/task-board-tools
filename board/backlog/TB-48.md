# TB-48: AgentService.CancelRun: SIGTERM/SIGKILL, JSONL cancelled, AgentStatus cancelled

**Type:** feature
**Priority:** P1
**Size:** M
**Module:** gui
**Tags:** milestone-m4,agent,wails-service,cancellation
**Branch:** —
**Parent:** TB-4

## Goal

User-initiated cancel. The three writes (kill, JSONL `cancelled`, `AgentStatus: cancelled`) happen in a fixed order so that even if the GUI crashes between steps, M5 stale-recovery sees enough to leave the task as `cancelled` rather than reinterpreting it as `failed`.

## Acceptance Criteria

- [ ] `AgentService.CancelRun(ctx, id) error` — looks up the in-flight `activeRun` entry from `AgentService.active` (the struct defined in TB-47); returns `ErrNotRunning` if no entry
- [ ] Cancellation executes in **this exact order** on the cancelling goroutine:
  1. **Mark cancelled**: under `activeRun.mu`, set `Cancelled = true`. This is what tells the TB-47 post-run handler to skip its own `finished`+Wails+AgentStatus writes
  2. **Kill**: call `cmd.Process.Signal(syscall.SIGTERM)` (using `activeRun.Cmd` directly, not the runner's ctx-cancel — explicit signal so the order is deterministic). Wait up to 5 seconds for `activeRun.Done` to close. If it doesn't close in time, `syscall.Kill(-activeRun.Pgid, syscall.SIGKILL)` so the whole process group dies; then wait on `Done` again with no further timeout
  3. **JSONL**: append `finished{ts, run_id, task_id, status: cancelled, reason: "user cancelled"}` via the TB-45 writer
  4. **Wails emit**: emit `agent:run-finished{run_id, task_id, status: cancelled, exit_code: -1}` so TB-52's store updates immediately
  5. **Metadata**: write `**AgentStatus:** cancelled` via `tb edit --agent-status cancelled`. This is the LAST step so that even if the GUI crashes between 4 and 5, the JSONL `cancelled` event is durable and M5 stale-recovery uses it to set `AgentStatus: cancelled` rather than `failed`
- [ ] Remove the `activeRun` entry from `AgentService.active` after step 5
- [ ] Idempotent: a second `CancelRun` for the same task returns `ErrNotRunning` cleanly (the entry is gone)
- [ ] Integration test (load-bearing — this is the F4.4 acceptance):
  - Start a stub agent that sleeps 60s and itself spawns a child sleep process; call `CancelRun`; assert (a) both processes exit within ~6s (PID-liveness probe via `syscall.Kill(pid, 0)`), (b) last JSONL line is `finished{status: cancelled, reason: "user cancelled"}`, (c) `tb show <ID>` reports `AgentStatus: cancelled`, (d) exactly one `finished` line exists in the JSONL for that `run_id` (the TB-47 post-run handler was suppressed)
  - Then construct a fresh `AgentService` (simulates restart); confirm `AgentStatus` is still `cancelled` and that the service's startup logic doesn't flip it. This seeds the M5 carve-out (TB-5's stale-recovery skips tasks whose `AgentStatus: cancelled`)
