# TB-109: Worktree-isolated task execution

**Type:** feature
**Priority:** P1
**Size:** L
**Module:** cli
**Tags:** epic
**Branch:** —

## Goal

Every code-producing task runs in its own git worktree on a dedicated `task/<ID>` branch. On completion the work is rebased onto the configured merge target and fast-forwarded back; the worktree and branch are cleaned up. On conflict the worktree is preserved for manual resolution. Foundation for parallel agent execution and the future epic fan-out / `max_workers > 1` story.

## Context

Discovered during brainstorming on epic agent orchestration: parallel agents on the same working tree fight over branches, build state, and uncommitted edits. Worktree isolation is the prerequisite that makes everything downstream (epic-level parallelism, GUI multi-agent runs) safe.

Spec: `docs/superpowers/specs/2026-05-14-worktree-isolation-design.md` (516 lines, eight rounds of Codex adversarial review).

Key invariants the spec preserves:

- **Board is canonical, lives only in main checkout.** Worktrees carry code only.
- **Three locks** (board / per-task / merge) with strict ordering.
- **Phase-tagged sidecars** make every crash point recoverable.
- **No blanket pre-checks** beyond what git itself enforces — the FF-finalize splits A/B/C on which worktree has `<merge_target>` checked out.
