# TB-39: Make CLI task-section scans ignore quoted Markdown headings

**Type:** bug
**Priority:** P2
**Size:** M
**Module:** cli
**Tags:** robustness,parser,markdown,section-headings
**Branch:** —

## Goal

Prevent CLI task mutations and triage checks from treating literal `## ...` text inside paragraphs, inline code, or fenced code blocks as real task-section headings.

## Context

The original repro was found through `tb create --parent`: `cmdCreate` holds `.board.lock`, writes the child with `writeFileAtomic`, then calls `addChildToSubtasks` to update the parent. The lock and atomic-write path are already the right invariants; the bug is the section scan inside `cli/create.go`.

Current `addChildToSubtasks` uses raw substring searches:

- `strings.Index(content, "## Subtasks")` to find an existing subtasks section
- `strings.Index(afterSubtasks, "\n## ")` to find the next section
- `strings.Index(content, "## Acceptance Criteria")` / `"## Log"` to choose where to create `## Subtasks`

That means a parent body like this can be corrupted when a child is created:

```markdown
## Goal

When grooming, do not edit `## Acceptance Criteria` directly.

## Acceptance Criteria

- [ ] Real criterion
```

Repro shape:

1. Create or fixture a parent task whose body mentions `` `## Acceptance Criteria` `` before the real `## Acceptance Criteria` heading.
2. Run `tb create "Child" -m cli -s S --parent TB-100`.
3. Today, `## Subtasks` can be inserted before the quoted text in the paragraph.
4. Expected: only the real section heading is used, so `## Subtasks` lands immediately before the actual `## Acceptance Criteria` heading; all other parent bytes are preserved.

The same raw heading-scan pattern exists in `cli/move.go:appendLogEntry` (used by `tb mv`, `tb start`, `tb done`, and `tb edit` log entries) and `cli/triage.go:hasPlaceholderSection` (used by `tb triage`). Fix the CLI heading-boundary helper once and use it at those section-scan sites.

## Parser / Heading-Scan Boundary

This task does **not** need a full Markdown parser. It needs a tiny task-section scanner for the CLI.

A structural task-section heading for this repo is:

- a line that begins at column 0 with `## `
- outside a fenced code block opened by a column-0 ``` or `~~~` fence
- exact heading text match after trimming trailing spaces, for example `## Acceptance Criteria` matches, but `## Acceptance Criteria draft`, `### Acceptance Criteria`, `- ## Acceptance Criteria`, and `` `## Acceptance Criteria` `` do not

The helper should return byte offsets or section bounds so callers can preserve all unrelated content byte-for-byte. Inline code is handled by the column-0 rule; fenced code needs explicit in-fence tracking because a code block may contain a column-0 `## Acceptance Criteria` line.

## Acceptance Criteria

- [ ] Add a small CLI section-heading helper (or equivalent local helper) that finds task-section headings and next-section boundaries using the boundary above; do not introduce a full Markdown/CommonMark dependency
- [ ] `addChildToSubtasks` uses the helper for `## Subtasks`, `## Acceptance Criteria`, and `## Log`; with no existing `## Subtasks`, it inserts the new section before the real `## Acceptance Criteria`, otherwise before the real `## Log`, otherwise at EOF
- [ ] Existing `## Subtasks` behavior is preserved: append the new child entry at the end of that section, before the next real `## ` section heading, without being fooled by quoted or fenced headings inside the Subtasks body
- [ ] Repro test: parent fixture contains `` `## Acceptance Criteria` `` in body text before the real heading; creating a child with `--parent` or calling the helper through `addChildToSubtasks` leaves the quoted text untouched and inserts `## Subtasks` before the real heading
- [ ] Fenced-code test: a parent body contains a fenced block with a line `## Acceptance Criteria`; the scanner ignores it and uses the real heading after the fence
- [ ] `appendLogEntry` in `cli/move.go` uses the same structural boundary so move/edit log entries are appended to the real `## Log` section and do not split on quoted or fenced `## ` text
- [ ] `hasPlaceholderSection` in `cli/triage.go` uses the same structural boundary so `tb triage` does not treat quoted headings as real sections or stop a real section at a fenced `## ` line
- [ ] Mutation invariants remain unchanged: structured CLI paths still hold `.board.lock`, task `.md` writes still go through `writeFileAtomic`, and `BOARD.md` is regenerated by the existing command flow
- [ ] `cd cli && go test ./...` passes

## Non-Goals

- Do not implement TB-75 (`tb edit --goal` / `--acceptance`) here; that task should reuse or follow the scanner contract from this fix
- Do not rewrite task files, normalize Markdown, or reorder sections beyond the specific insertion/log/triage changes

- Do not change GUI direct-write helpers (`gui/app/edit_body.go:protectedPrefix` / `appendBodyEditLog`) in this task; file a follow-up if the same bug is confirmed in the GUI surface
