# TB-120: Per-task MergeTarget override

**Type:** feature
**Priority:** P1
**Size:** S
**Module:** cli
**Branch:** —
**Parent:** TB-109

## Goal

Add a per-task `**MergeTarget: <branch>**` metadata field that overrides the board-level `worktrees.merge_target` default. Parsing in `parseTaskFile`, a `tb edit --merge-target` flag to set it, and a single `resolveMergeTarget(task)` helper that all worktree-flow code (TB-113 / TB-115 / TB-116 / TB-117) consults.

## Context

Spec sections: § 4.3 (base branch), § 5 (new task metadata fields), § 11 decision 5.

Without this, every task is forced to merge into the same `<merge_target>` even when a feature needs to go to e.g. `release/v2` instead of `main`. Cheap to add now, painful to retrofit later because every flow that resolves the target would need updating.

## Acceptance Criteria

- [ ] `parseTaskFile` (`cli/task.go`) recognizes `**MergeTarget:** <branch>` in the first 15 lines and exposes it as `Task.MergeTarget` (empty string when absent).
- [ ] `tb edit <ID> --merge-target <branch>` sets/updates the field. Empty value (`--merge-target=""`) clears it back to "use board default".
- [ ] `tb show <ID>` and `tb show --json <ID>` surface the field when present.
- [ ] A single helper `resolveMergeTarget(task, boardConfig)` returns: task's `MergeTarget` if set, else `boardConfig.Worktrees.MergeTarget`, else hard default `main`.
- [ ] TB-113 (`tb start`), TB-115 (`tb done` step 9 case A/B/C and step 5 rebase), TB-116 (`tb merge` re-rebase), and TB-117 (recovery's `mergeContainsBranch` helper) all consume the resolver — no flow hard-codes `<merge_target>` from board config.
- [ ] The sidecar (`<board>/.worktrees/<ID>.json`) records the resolved `mergeTarget` at creation time so recovery doesn't need to re-resolve against possibly-changed task metadata.
- [ ] Integration test: create a task with `**MergeTarget:** release`, run `tb start` → `tb done`, verify FF goes to `release` not `main`.
