# TB-60: Stale-running recovery: scan AgentStatus=running, JSONL replay, synthetic finished+failed

**Type:** feature
**Priority:** P1
**Size:** M
**Module:** gui
**Tags:** milestone-m5,agent,daemon,recovery
**Branch:** —
**Parent:** TB-5

## Goal

On daemon start, before the queue scan (TB-57), find every task with `AgentStatus=running`. For each, read the last run from `board/.agent-state/<ID>.jsonl`; if there is no `finished` event AND the PID from the `started` event is dead (verified via `pidAlive` TB-59), write a synthetic `finished{status: failed, reason: "stale after restart"}` JSONL line and flip `AgentStatus` to `failed` via `tb edit`.

## Context

R5 / R12 / F5.2. The crash story: GUI is running an agent → process gets `kill -9` (or power loss). On disk: the task `.md` shows `AgentStatus: running`, the `.jsonl` has a `started` event but no `finished`. Without recovery, the task stays "running" forever and the user has to manually fix it. Recovery reconciles by inferring the run died.

Algorithm:

1. `BoardService.LoadBoard("active")` → list of tasks where `AgentStatus=running`.
2. For each task:
   a. Read `board/.agent-state/<ID>.jsonl` line-by-line; find the latest run (by `started`+`run_id`). The `queued` event for that `run_id` carries the `agent` field; the `started` event carries only `pid`. **The `expectedName` for the pid-check comes from the `queued` event** (or from TB-54's schema change that adds `agent` to `started` — pick one source and document it).
   b. If a `finished` event exists for that `run_id` → JSONL is fully written but `.md` was never updated (rare: AgentStatus write failed). Set `AgentStatus` to whatever the `finished{status}` says.
   c. Otherwise check `pidAlive(pid, expectedAgent)` (TB-59):
      - alive → leave alone (someone else's process or our own re-attach; don't kill — no re-attach in M5).
      - dead → append synthetic `finished{status: "failed", reason: "stale after restart", recovered_at: <now>}` line, then `tb edit <ID> --agent-status failed`.
3. **Carve-out** (TB-61): if latest event is `finished{status: cancelled}`, reconcile to `cancelled`, not `failed`. Never overwrite `AgentStatus=cancelled`.
4. Log each recovery decision at INFO with task_id + outcome.

Ordering: recovery **before** queue scan. Reason: a recovered task is now `failed`, not `queued`, so the subsequent scan does not pick it up. If the order were reversed, a future change that bumps recovered tasks back to `queued` would cause double work.

Note: the cancel-mid-crash carve-out (TB-61) is a separate task because the test requires a specific kill-9-mid-cancel scenario.

## Acceptance Criteria

- [ ] New function `Daemon.recoverStale(ctx)` runs before the queue scan
- [ ] Lists tasks via `BoardService.LoadBoard("active")` where `AgentStatus=running`
- [ ] Reads `board/.agent-state/<ID>.jsonl`, locates the latest `run_id`'s events; pulls `agent` from the `queued` event (or `started` if TB-54's schema update lands it there) and `pid` from `started`
- [ ] If `finished` exists with status `success|failed`, sync `AgentStatus` to match via `tb edit --agent-status`
- [ ] If `finished` is missing AND `pidAlive(pid, expectedAgent)==false`: append `finished{status: "failed", reason: "stale after restart", task_id, run_id, recovered_at}` via `state.AppendEvent`, then `tb edit --agent-status failed`
- [ ] If `pidAlive==true`: log INFO `live PID detected, skipping recovery` and leave the task alone — M5 does **not** re-attach to in-flight output; conservative default
- [ ] **F5.2 acceptance 1 (Go unit/in-process fixture)**: stage a tempdir board with a task having `AgentStatus=running`, a JSONL with `queued`+`started{pid: dead_pid}` and no `finished`. Run `recoverStale` against it; assert synthetic `finished{failed, "stale after restart"}` line appended and `AgentStatus=failed`.
- [ ] **F5.2 acceptance 1 (multi-process harness)**: separately, a manual / scripted scenario kills the running GUI with `kill -9` while an agent is mid-flight and restarts the GUI; documented in `gui/internal/daemon/README.md` as a manual smoke step rather than a CI gate — Go test runtime can't reliably reproduce parent-process death.
- [ ] Recovery emits Wails event `agent:run-finished` so any open drawer updates immediately (parity with M4 normal exit)
