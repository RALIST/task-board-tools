## Goal

Set up the SvelteKit shell under gui/frontend: typed Wails bindings in lib/api.ts, board+filter stores in lib/stores/, and the single route +page.svelte that subscribes to board/task events and re-fetches via api.LoadBoard. Empty visual state — components land in TB-23.

## Context

Wires the frontend to the Go services so subsequent component work (TB-23, TB-24) only deals with rendering. `api.ts` is a thin typed layer over the Wails-generated bindings so the rest of the frontend never imports Wails directly — that lets us swap in a mock for storybook/tests later. Stores wrap the snapshot and merge per-task updates from `task:updated:<id>` events without re-fetching the whole board.

## Acceptance Criteria

- [ ] `gui/frontend/src/lib/api.ts` exposes typed `loadBoard()`, `getTask(id)`, `openBoard(root)`, `pickBoardDialog()`, `listRecentBoards()` wrapping the Wails bindings
- [ ] `gui/frontend/src/lib/stores/board.ts` — Svelte store holding the latest `BoardSnapshot`; `task:updated:<id>` events patch a single task without re-fetching
- [ ] `gui/frontend/src/lib/stores/filter.ts` — Svelte store with placeholder filter state (consumed in M3 by FilterBar)
- [ ] `gui/frontend/src/routes/+page.svelte`: on mount calls `loadBoard()`, subscribes to `board:reloaded` and `task:updated:<id>`, renders a placeholder shell (real components land in TB-23)
- [ ] Empty-state UI when no board is selected — call to action that triggers `pickBoardDialog`
- [ ] Error toast surface for service errors (a minimal Toast component is fine — M3 will expand it)
- [ ] `wails3 dev` renders the shell against the live board with zero console errors
