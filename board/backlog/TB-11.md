# TB-11: Add Agent and AgentStatus task metadata fields

**Type:** feature
**Priority:** P1
**Size:** M
**Module:** cli
**Tags:** milestone-m1,agent,metadata
**Branch:** —
**Parent:** TB-1

## Goal

Add two optional metadata fields (Agent, AgentStatus) to task .md files. Parse them in cli/task.go, expose them on the Task struct with JSON tags, and add -a / --agent-status flags to tb edit. AgentStatus values: queued | running | success | failed | cancelled.

## Context

Two new optional metadata fields on every task: `Agent` (`claude` | `codex` | empty) and `AgentStatus` (`queued` | `running` | `success` | `failed` | `cancelled` | empty). Existing tasks without these fields must still parse — empty is the default and the line should not be rendered for empty values to keep old task files clean. `tb edit` gains `-a` and `--agent-status` flags. `AgentStatus=cancelled` is user-initiated and stale-recovery (M5) must never overwrite it. Optional sugar: `tb assign <ID> <agent>` as shorthand for `tb edit <ID> -a <agent> --agent-status queued`.

## Acceptance Criteria

- [ ] `Task` struct gains `Agent string \`json:"agent"\`` and `AgentStatus string \`json:"agentStatus"\``
- [ ] `parseTaskFile` (`cli/task.go:30`) extracts both via `extractFieldAny`
- [ ] `cli/edit.go`: `-a`/`--agent` and `--agent-status` flags wired through `setField`
- [ ] `tb create` does not emit the lines unless the values are non-empty
- [ ] `tb edit 1 -a claude --agent-status queued` round-trips through `tb show 1` and `tb ls --json`
- [ ] `--agent-status` validates against the enum; invalid value fails with a clear error
- [ ] (Optional) `tb assign <ID> <agent>` sugar exists
