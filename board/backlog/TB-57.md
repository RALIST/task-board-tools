# TB-57: Daemon initial queue scan on startup (AgentStatus=queued → enqueue)

**Type:** feature
**Priority:** P1
**Size:** S
**Module:** gui
**Tags:** milestone-m5,agent,daemon,startup-scan
**Branch:** —
**Parent:** TB-5

## Goal

After the daemon starts (and the stale-recovery scan TB-60 has finished — order matters), enumerate all active-status tasks via `BoardService` and enqueue any with `AgentStatus=queued` into the daemon's worker pool. This catches tasks that were queued from the CLI between GUI sessions.

## Context

Watcher events (TB-58) only fire on *future* file changes. Tasks queued via `tb edit <ID> --agent-status queued` while the GUI was closed must be picked up on next launch — that is the job of this scan.

Strict startup ordering required:

1. **Stale-running recovery (TB-60)** — flips dead `running` tasks to `failed` (or `cancelled` per TB-61). Must complete before scan so the scan reads a consistent post-recovery view.
2. **Watcher event sink registration (TB-58)** — must be live before the scan so that an event arriving between scan-read and scan-finish is captured rather than dropped. Without this ordering, a CLI edit that lands `between` (a) recovery finishing and (b) scan completing can be missed: the file change arrives, no subscriber is listening, the scan reads the pre-edit snapshot, the edit's event is gone.
3. **Startup queue scan** — this task.

Active-set dedup (TB-55) tolerates the overlap where both step 2's event and step 3's scan see the same task — only the first `tryEnqueue` wins.

Use the in-process `BoardService.LoadBoard("active")` rather than shelling to `tb ls --json` — same data, no fork, and consistent with the GUI's read path.

Status filter: `active` (backlog + in-progress + done) is the right scope. A `done` task with `AgentStatus=queued` is weird but legal — the daemon should still run it (or alternatively, log + skip). Pick **run it** for symmetry with the watcher pickup path.

## Acceptance Criteria

- [ ] Strict startup order: (1) `recoverStale` (TB-60) returns → (2) watcher event sink (TB-58) is registered and reading → (3) startup scan runs. Document this in `daemon.Start` with comments.
- [ ] Scan uses `BoardService.LoadBoard("active")` (not shell-out to `tb ls`)
- [ ] For each task with `AgentStatus=queued` and an assigned `Agent`, call `Daemon.tryEnqueue(taskID)` (TB-55)
- [ ] Tasks without `Agent` set but with `AgentStatus=queued` are logged at WARN and skipped
- [ ] **F5.1 acceptance**: from a terminal, run `tb edit X --agent-status queued -a claude` while GUI is closed; launch GUI; within 5s the task transitions to `running` (verified by `tb show X` or the GUI)
- [ ] Unit test: load a synthetic board with mixed AgentStatus values; assert only the `queued` ones are enqueued
- [ ] Scan logs `slog.Info` with count of enqueued tasks for observability
