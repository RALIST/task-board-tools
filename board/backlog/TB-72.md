# TB-72: TaskDrawer: Groom button next to Run, shared Cancel, mode-labelled past runs

**Type:** feature
**Priority:** P2
**Size:** M
**Module:** gui
**Tags:** milestone-m6,frontend,drawer,groom
**Branch:** —
**Parent:** TB-6

## Goal

Add a **Groom** button next to **Run** in `TaskDrawer.svelte`. Enabled when an agent is assigned AND no run is in flight. Click → `groomTask(id)` (TB-71) → identical streaming + Cancel UX. Past-runs list (TB-52) renders a `groom` / `implement` mode tag next to each row.

## Context

The drawer's existing Run/Cancel section (`onRunClick`, `startCancel`, `selectedRun`, `liveStatus`) is mode-agnostic at the UI layer — it talks about run IDs and `agent:run-*` events. Adding Groom is a parallel call site that shares state with Run (same `runsStore`, same `selectedRunID`, same Cancel button). The button position should mirror the docs ("next to Run") and reuse the existing `Toast.svelte` for errors. Disabled state: same predicate as Run (agent assigned, not running) — visually marked.

## Acceptance Criteria

- [ ] `TaskDrawer.svelte` has a `Groom` button next to `Run`; same shape and disabled-state logic
- [ ] Click → `await groomTask(id)`; on success upserts a Run record with `mode='groom'`, `status='queued'` and sets `selectedRunID` (mirrors `onRunClick`)
- [ ] On `groomTask` error: pushes a toast (`Groom failed: …`); no state mutation
- [ ] Cancel button cancels whichever run is active (groom or implement) — no new API surface needed
- [ ] Past-runs row renders the run's mode as a small label / pill next to the timestamp (`groom` / `implement`)
- [ ] When `triageStore.needsGrooming(currentTaskID)` is true and no run is in flight, the Groom button gets a subtle visual highlight (`ring` or accent border) so the indicator-click path from TB-73 has somewhere to point
- [ ] Vitest covers: button disabled with no agent / while running; click calls `groomTask`; toast on error; selected run mode label rendered
- [ ] No backend behaviour asserted here — that's TB-67 / TB-68's job
