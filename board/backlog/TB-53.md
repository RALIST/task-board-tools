# TB-53: Daemon skeleton + Wails OnStartup/OnShutdown lifecycle wiring

**Type:** feature
**Priority:** P1
**Size:** S
**Module:** gui
**Tags:** milestone-m5,agent,daemon,skeleton
**Branch:** —
**Parent:** TB-5

## Goal

Create `gui/internal/daemon` package with a `Daemon` struct, `New`/`Start(ctx)`/`Close()` lifecycle, and wire it into Wails `App.OnStartup` / `App.OnShutdown` in `gui/main.go` so a single daemon goroutine runs while the GUI is open.

## Context

The daemon is the M5 backbone — every other M5 sub-task hangs off it. Before any feature wiring (queue scan, watcher subscription, recovery), we need a package skeleton with the lifecycle surface that the rest of M5 fills in.

Crucially, the daemon has **two lifecycle phases**, not one — because the GUI starts before a board is opened:

1. **Construction** (in `main`, before `app.Run()`): the daemon is created with handles to BoardService, AgentService, watcher, logger. Workers are spawned and idle. No file IO, no scan, no recovery.
2. **Activation** (when `SettingsService.OpenBoard` succeeds): `BoardService` gets a CLI client targeting the project root, and `watcher.Switch(boardDir)` is called. Only then does the board path exist for the daemon to scan / recover against. The daemon registers an `OnBoardOpen(boardDir)` callback (or `SettingsService` exposes a hook) that drives `Daemon.Activate(boardDir)` → recovery (TB-60) → watcher-sink registration (TB-58) → startup scan (TB-57).

Re-activation: if the user opens a different board later, `Daemon.Deactivate()` must drain workers and clear active-set state before `Activate(newBoardDir)` runs again. The single-instance lock (TB-2) means only one board is active at a time per user.

`Close()` is called from the shutdown path with a bounded grace window (the 5s wait lives in TB-62; this task only wires the call).

## Acceptance Criteria

- [ ] New file `gui/internal/daemon/daemon.go` (package `daemon`) defines:
  - `type Daemon struct` whose fields are narrow interfaces (`Board interface { LoadBoard, GetTask }`, `Agent interface { RunQueuedAgentSync, HasActiveRun, ... }`, `Logger *slog.Logger`) — tests use fakes that implement the interfaces, not the concrete services
  - `type Options struct { Board, Agent, Logger, MaxWorkers }` — `MaxWorkers` defaulted to 1 when zero or out of range
  - `func New(opts Options) *Daemon` — constructs, starts worker goroutines in idle state, no IO
  - `func (d *Daemon) Activate(ctx context.Context, boardDir string) error` — recovery (TB-60) → watcher sink registration (TB-58) → startup scan (TB-57). Caller is `SettingsService.OpenBoard` (or a hook registered from `main.go`).
  - `func (d *Daemon) Deactivate() error` — drains workers, clears active-set, closes the watcher sink subscription. Called when the user switches boards.
  - `func (d *Daemon) Close() error` — calls Deactivate, cancels root ctx, returns. 5s grace lives in TB-62.
- [ ] `gui/main.go` constructs the daemon before `app.Run()` and registers `daemon.Activate` as a `SettingsService.OpenBoard` post-hook (or `SettingsService` calls into `daemon` directly via a `BoardActivator` interface)
- [ ] Daemon does **not** scan or recover before `Activate` is called — fresh GUI launch with no recent board is a no-op
- [ ] Daemon respects the M2 single-instance lock (TB-2) inherited from `gui/main.go`'s `SingleInstance` config
- [ ] `go test ./gui/internal/daemon/...` exercises New/Activate/Deactivate/Close lifecycle against fake `Board`/`Agent` implementations (no concrete services)
- [ ] No regression in existing GUI tests (`go test ./gui/...`)
