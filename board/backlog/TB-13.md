# TB-13: Call regenerateBoard at end of create and edit

**Type:** bug
**Priority:** P1
**Size:** S
**Module:** cli
**Tags:** milestone-m1,regenerate,quick-win
**Branch:** —
**Parent:** TB-1

## Goal

tb create and tb edit currently skip regenerateBoard, so BOARD.md drifts from the directory state until a manual tb regenerate. Add the call at the end of cmdCreate and cmdEdit so every structured mutation refreshes BOARD.md and fires a single fsnotify signal the GUI can subscribe to.

## Context

`tb mv`, `start`, `done`, `close`, `scan` all call `regenerateBoard` at the end. `tb create` and `tb edit` don't (`cli/create.go:127`, `cli/edit.go:116`), so `BOARD.md` can lag the directory state and the single fsnotify signal the GUI relies on never fires. One-liner fix at the end of `cmdCreate` and `cmdEdit`.

## Acceptance Criteria

- [ ] `cmdCreate` (`cli/create.go`) calls `regenerateBoard(boardDir)` after a successful write
- [ ] `cmdEdit` (`cli/edit.go`) calls `regenerateBoard(boardDir)` after a successful write

- [ ] `tb edit X-N -p P1` updates `BOARD.md` without a separate `tb regenerate`
- [ ] Existing tests still pass
