## Goal

Ship a first-class Code Review workflow for completed implementation work: tasks can move into a `code-review` column with implementation refs and reviewer notes, reviewers can record findings, and failed reviews return to backlog as visibly prioritized rework.

## Context

- Board statuses are directory based; adding `code-review` means changing the CLI status taxonomy, generated board output, GUI board buckets, watcher refresh paths, and docs together.
- Review metadata should live in dedicated task sections: `## Review Target`, `## Reviewer Notes`, and `## Review Findings`.
- Implementation refs can be a branch, PR URL, commit SHA, worktree path, or a short free-form note that tells a reviewer where to inspect the change.
- Existing agent mode plumbing for implement/groom runs is the model for review mode; review runs should reuse the same JSONL, Wails, daemon, cancellation, and recovery paths.
- Related automation epics: TB-177/TB-179 for auto-implementation priority and TB-182 for a separate user-attention protocol.
- This is an epic. Ship via the child tasks below; do not try to land the whole workflow in one change.

## Constraints / Non-goals

- Preserve board invariants: structured mutations take `.board.lock`, task writes are atomic, and `BOARD.md` is regenerated by `tb` commands.
- `code-review` is a board status/column, not an `AgentStatus` value.
- `review-failed` is a tag for backlog rework, not a new status and not the same as `needs-user` from TB-182.
- Missing Review Target on submit should warn for the MVP, not hard-fail.
- Review Findings must be actionable and live in the ticket so humans and agents can pick up the next step without reading private run logs.
- GUI work must include a manual smoke note because this epic changes visible kanban behavior.

## Acceptance Criteria

- [ ] **TB-195** is done: CLI/status support includes the `code-review` active status, aliases, JSON/board output, and submit warning behavior.
- [ ] **TB-196** is done: managed CLI commands can write Review Target, Reviewer Notes, and Review Findings sections without hand-editing markdown.
- [ ] **TB-197** is done: the GUI renders Code Review as an active column and surfaces review metadata/failure markers on cards and in the drawer.
- [ ] **TB-198** is done: users can run review-mode agents and review findings land in `## Review Findings` through managed board commands.
- [ ] **TB-199** is done: failed reviews move from Code Review back to Backlog with `review-failed`, visible findings, and retry-priority behavior.
- [ ] **TB-200** is done: board conventions, agent rules/prompts, and product/architecture docs describe the complete human and agent workflow.
- [ ] End-to-end happy path: create/start an implementation task, set Review Target and Reviewer Notes, submit it to Code Review, run or perform review, record no blocking findings, and move it to Done.
- [ ] End-to-end failed path: submit a task to Code Review, record actionable Review Findings, fail it back to Backlog with `review-failed`, then resubmit after fixes with the agreed marker-clear behavior.
- [ ] Manual test note: exercise CLI submit/fail commands plus GUI drag/drop into Code Review, drawer rendering of all review sections, review-mode run history, failed-review marker visibility, and final move to Done.
- [ ] Epic verification includes `cd cli && go test ./...`, `cd gui && go test ./...`, `cd gui/frontend && npm run check`, and `cd gui/frontend && npm test -- --run`.
