# TB-66: GroomingDecorator: wrap a Runner, swap prompt template

**Type:** feature
**Priority:** P2
**Size:** M
**Module:** gui
**Tags:** milestone-m6,agent,runner,decorator
**Branch:** —
**Parent:** TB-6

## Goal

Add `agent.GroomingDecorator` — a `Runner` that wraps a concrete `ClaudeRunner` / `CodexRunner`, captures a `PromptVars` value at construction time, and on `Run` overwrites `RunInput.Prompt` with `RenderPrompt(PromptGroom, vars)` before delegating. Reports the same `Name()` as the inner runner. The decorator is the only layer aware of `ModeGroom`; the inner runners stay mode-agnostic.

## Context

`Runner.Run(ctx, in RunInput)` receives an already-rendered `Prompt` string — `RunInput` deliberately carries no `PromptVars`. Today the call site in `gui/app/agent_run.go` renders `PromptImplement` against fresh `PromptVars{TaskID, TaskTitle, TaskBody}` just before invoking the runner. For the groom path the decorator needs the same vars to render `PromptGroom`. Options considered:

- **Add `RunInput.PromptVars`** and re-render inside the decorator — pollutes `RunInput` for a single mode, breaks the "caller renders, runner consumes" contract.
- **Capture vars at decorator construction** via `NewGroomingDecorator(inner Runner, vars PromptVars)` — keeps `RunInput` schema-stable, the AgentService caller already has the vars at the point it builds the decorator (right after `BoardService.GetTask`), and the daemon path (TB-68) has them too via the same `GetTask` lookup. **Pick this.**

The decorator does not need to inspect `in.Mode` — its existence implies groom. The caller (TB-67 / TB-68) only constructs it when mode is groom.

## Acceptance Criteria

- [ ] `agent.GroomingDecorator` implements `Runner`; `Name()` returns the inner runner's name (so JSONL / Wails events still read `claude` / `codex`, not `groom`)
- [ ] `Run(ctx, in)` rewrites `in.Prompt` to `RenderPrompt(PromptGroom, vars)` (using the vars captured at construction) before delegating; no other `RunInput` field is mutated
- [ ] `NewGroomingDecorator(inner Runner, vars PromptVars) Runner` is the only public surface; the struct stays unexported
- [ ] Unit test with a stub inner Runner: asserts the inner receives the rendered groom prompt verbatim (placeholders substituted from the constructor vars) AND the original `TaskID/ProjectRoot/Env/Timeout/OnStarted` are forwarded unchanged
- [ ] Unit test asserts the caller's pre-rendered `in.Prompt` (e.g., the implement template the agent_run path passes today) is overwritten — the decorator does not concatenate or fall back to it
- [ ] Unit test confirms `Run` returns the inner runner's `RunResult` and error untouched (decorator does not add error-wrapping)
- [ ] No change to `claude.go`, `codex.go`, or any other Runner — they stay mode-agnostic
- [ ] No change to `RunInput` — fields stay byte-identical (verified by an interface-stability test or by inspection in review)
- [ ] `defaultRunnerFactory` is NOT touched here (M4 contract preserved); TB-67 owns the wrap-on-the-way-out call site
