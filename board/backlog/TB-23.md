## Goal

Implement Board.svelte (3-column grid), Column.svelte (header + scrollable list), and Card.svelte (compact: ID, title, priority pill, type icon, tags chips, agent badge, epic marker). Read-only — no DnD yet (M3). Reactive to board store updates from fsnotify events.

## Context

The visible deliverable for M2. Three columns from a single store; no DnD (M3) and no archive column by default (the plan keeps archive behind a toggle and ships the toggle itself in M3 via FilterBar). Card density should be high enough that ~30 tasks fit on a typical 1440px window without scroll. CLAUDE.md mandates using the `frontend-design` skill for this kind of work — the implementer should invoke it before writing the components.

## Acceptance Criteria

- [ ] `Board.svelte` lays out three columns (backlog / in-progress / done) with a CSS grid that flexes from 1024px upward
- [ ] `Column.svelte` shows header (title + count) and a scrollable card list
- [ ] `Card.svelte` displays: ID prefix, title (truncated to 2 lines), priority pill (P0..P3 color-coded), type icon, tag chips (max 3 visible + "+N"), agent badge when `agent` is set, epic marker when `tags` contains `epic`
- [ ] Click on a card emits an event that `+page.svelte` translates into opening TaskDrawer (TB-24)
- [ ] Components consume the board store — no direct API calls
- [ ] Renders cleanly on an empty board (empty columns, no errors)
- [ ] Cards re-render in place when the corresponding `task:updated:<id>` event fires (no full board reload required)
- [ ] Implementer follows the `frontend-design` skill for visual quality (per CLAUDE.md instruction)
