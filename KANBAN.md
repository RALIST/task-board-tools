Core columns (minimum viable board):
1. Backlog — requested, not yet committed
2. To Do / Ready — committed, prioritized, pullable
3. In Progress / Doing — actively worked
4. Done — finished

Often expanded to: Backlog → Ready → In Progress → Review → Test → Done, with explicit "Doing / Done" sub-columns per stage to make handoffs visible.

The five practices:
1. Visualize the workflow — every work item on a card, every state as a column.
2. Limit WIP — explicit numeric cap per column (e.g. "In Progress: 3"). This is the defining constraint.
3. Manage flow — measure cycle time and throughput; smooth bottlenecks.
4. Make policies explicit — entry/exit criteria written on the board.
5. Improve collaboratively — evolve the system with feedback loops.

Core mechanics:
- Pull, don't push — a downstream column pulls when it has capacity, rather than upstream pushing work in.
- One card = one work item — moves left-to-right, never skips columns, never moves backward except for explicit rework.
- Cards carry — title, owner, age/start-date, class of service (standard / expedite / fixed-date / intangible).
- Cumulative Flow Diagram is the canonical metric view; Little's Law (WIP = throughput × cycle time) is the underlying math.

The non-negotiables are visualize + WIP limits + pull. Everything else is variation.
